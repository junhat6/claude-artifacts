影は、下にしか落ちない。

机に貼った付箋は、宙に浮かない。tldraw の付箋がなぜ「本物っぽい」のか——その正体を分解して、この PoC に移植した。以下は、その熱弁。証拠は下のボードにある。まず触ってほしい。

クリック = 選択 もう一度 = 編集 ドラッグ = 移動 Backspace = 削除 Esc = 編集終了

影の解剖

「浮いたカード」と「貼られた紙」の違いは、影の向きだけ

UI の影はたいてい全方向に広がる。shadow-sm をそのまま使うと、要素は空中に浮かんだカードに見える。それ自体は悪くない。ただし付箋には致命的に合わない。付箋は浮いていない。机に貼られている。貼られた紙の影は、めくれた下端からしか漏れない。

tldraw の実装から移植したトリックはこうだ。box-shadow の spread に blur 以上のマイナス値を与えると、影は要素の輪郭より内側に畳み込まれて一旦消える。そこへ正の Y オフセットを足すと、下辺からだけ影が覗く。横にも上にも、影は一切はみ出さない。

  • ① 接地影濃く、短い。紙の下端が机に触れているきわの暗さ。浮くほど痩せて消える。
  • ② 拡散影薄く、遠い。めくれた下端が机に落とすぼやけた影。浮くほど遠く、柔らかくなる。
  • ③ 紙面の内側陰影ほぼ見えない濃さの inset。紙のわずかな質感。気づかれないことが仕事。
/* note-shadow.ts — 3層とも Y オフセットは正、spread は負 */
0px 4.35px 5px    -5px  rgba(15, 23, 31, 0.60)  /* ① 接地影 */
0px 8.55px 10.55px -7.9px rgba(15, 23, 31, 0.36)  /* ② 拡散影 */
0px 36px  10px   -10px rgba(15, 23, 44, 0.024) inset /* ③ 紙面 */

もうひとつ、見落とされがちな芸がある。浮き量は付箋の id から決定的に導出され、1枚ごとに違う。本物の付箋を10枚貼れば、めくれ方は10通りになる。全部の影が同一だと、その瞬間に「複製された UI 部品」に戻ってしまう。tldraw はこれを id シードの乱数でやっていて、この PoC もそのまま踏襲した。

そしてドラッグ中は浮き量が +1 される。接地影が痩せ、拡散影が伸びて薄まる——指でつまんで持ち上げた紙の影そのものだ。上のボードでドラッグして、影だけを見ていてほしい。

選択の合図

青い枠は「今からこれを触る」という宣言

ホバーでもフォーカスリングでもない。pointerdown の瞬間、指が触れたその時に青い枠が付く。この即時性が「摘まんだ」感覚を作る。枠の色は tldraw の選択色 hsl(214, 84%, 56%) をほぼそのまま使った。アンバーの紙に対する補色に近い青で、机の上でこれ以上ないほど目立つ。

選択は常に 1 枚だけ。別の付箋を押せば選択は移り、ボードの余白を押せば解除される。「いま自分がどれを操作対象にしているか」が常に画面上で一意に答えられる——モードを作るなら、モードの所在は見えなければならない。

状態設計

1回のクリックに、3つの意味を持たせる

旧実装の付箋には「ドラッグ用のつまみバー」と「削除ボタン」が付いていた。機能の数だけ UI 部品を足す、正直な設計だ。しかし tldraw は同じ機能をクリック 1 種類の使い分けで実現していて、付箋の顔には何も付いていない。

  1. 1回目のクリック — 選択pointerdown で即座に青い枠。この時点ではまだ何も壊れない。
  2. 選択済みへのもう1回 — 編集「pointerdown の時点で選択済みだったか」を覚えておき、移動なしの pointerup で編集開始。
  3. 押したまま 4px — ドラッグ閾値を超えた瞬間にドラッグへ昇格。超えるまでは、これはまだクリック。

鍵は 4px の閾値だ。人間のクリックは物理的に数 px 揺れる。閾値なしで実装すると、選択するつもりのクリックが毎回微小なドラッグとして飛び、編集に入りたいだけなのに付箋がじりじり動く。4px は tldraw のドラッグ判定距離で、「意図した移動」と「手の震え」の境界線として実証済みの値だ。

キーボードも同じ思想で、モードが曖昧さを解決する。Backspace は編集中なら文字を消し、選択中なら付箋を消す。同じキーの2つの意味が、状態の所在（青い枠か、カーソルの点滅か）で見分けられる。

キー選択中（青い枠）編集中（カーソル点滅）
Backspace / Delete付箋を削除文字を削除
Enter編集を開始改行
Esc—編集を終了して選択に戻る

設計メモ

移植で決めたこと

状態はどこに置くか

選択は「同時に 1 枚だけ」という付箋同士の関心事なので BoardView が持つ。編集はその付箋に閉じた関心事なので NoteCard のローカルに持つ。状態の置き場所を、関心の届く範囲に一致させる。

操作面はなぜ透明な button か

クリックとキー操作を受ける要素は意味的には button だが、button は対話的な子要素（textarea）を内包できない。そこで非編集時だけ本文の上に透明な実 button を重ねる。編集に入ると button ごと消えるので、ポインターも Backspace も自然に textarea へ渡る。このページのデモの付箋も、まったく同じ構造で動いている。
