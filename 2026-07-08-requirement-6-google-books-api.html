GoogleBooks クライアント（app/services）

新規ファイル app/services/google_books.rb ・ ランタイム gem 追加なし

外部 API との通信を 1ファイルのサービスクラスに隔離した。HTTP クライアントは Ruby 標準の Net::HTTP で、httparty や faraday のような gem は入れていない。呼び出し側（コントローラ）から見た結果はきっかり三値：ヒット＝Volume、ヒットなし＝nil、通信・API 異常＝GoogleBooks::Error。

ヒットVolume（値オブジェクト）正常系 → フォームに反映

Data.define(:title, :authors, :publisher, :published_year)。API の生 JSON をコントローラに漏らさず、必要な4項目だけに整形して返す。

「見つからない」は失敗ではない — 洋書や新刊は普通にヒットしない
ヒットなしnil正常系 → 手入力を促す

API は 200 を返したが該当書籍がなかった。呼び出し側は「見つかりませんでした」と案内する。例外にしない。

「壊れている」は別物 — タイムアウト・5xx・不正 JSON はここ
異常GoogleBooks::Error異常系 → rescue して案内

ネットワーク断・タイムアウト・HTTP エラー・JSON 破損をすべてこの1例外に集約。呼び出し側は rescue 1個で済む。

app/services/google_books.rb（公開 API 部分）ruby
class GoogleBooks
  Error = Class.new(StandardError)
  Volume = Data.define(:title, :authors, :publisher, :published_year)

  API_URL = "https://www.googleapis.com/books/v1/volumes".freeze
  TIMEOUT_SECONDS = 5

  class << self
    def lookup(isbn)
      normalized = normalize(isbn)
      return nil if normalized.empty?

      parse(request(normalized))
    end

    # ハイフン・空白の揺れを吸収する（"978-4-..." と "9784..." を同一視するため）
    def normalize(isbn)
      isbn.to_s.gsub(/[\s-]/, "")
    end
app/services/google_books.rb（HTTP と解析：private）ruby
def request(isbn)
  uri = URI(API_URL)
  uri.query = URI.encode_www_form(q: "isbn:#{isbn}")

  response = Net::HTTP.start(uri.host, uri.port, use_ssl: true,
                             open_timeout: TIMEOUT_SECONDS, read_timeout: TIMEOUT_SECONDS) do |http|
    http.get(uri.request_uri)
  end
  raise Error, "Google Books API がエラーを返しました（HTTP #{response.code}）" unless response.is_a?(Net::HTTPSuccess)

  JSON.parse(response.body)
rescue Timeout::Error, SystemCallError, SocketError, IOError, OpenSSL::SSL::SSLError => e
  raise Error, "Google Books API に接続できませんでした: #{e.message}"
rescue JSON::ParserError => e
  raise Error, "Google Books API の応答を解析できませんでした: #{e.message}"
end

def parse(json)
  info = json.dig("items", 0, "volumeInfo")
  return nil if info.nil? || info["title"].blank?

  Volume.new(
    title: info["title"],
    authors: Array(info["authors"]),
    publisher: info["publisher"],
    # "2012-06" や "1914" のような形式差があるため、先頭の4桁だけを年として使う
    published_year: info["publishedDate"]&.slice(/\d{4}/)&.to_i
  )
end

活用した Rails / Ruby の機能

  • Net::HTTP（標準ライブラリ）
    gem 追加なしで HTTPS + タイムアウト設定まで完結。依存が増えない＝アップデートで壊れる場所が増えない。
  • Data.defineRuby 3.2+
    イミュータブルな値オブジェクト。Struct と違い後から書き換えられないので「API の取得結果」の性質に合う。
  • Hash#dig
    json.dig("items", 0, "volumeInfo")。深いネストの途中が欠けていても例外にならず nil が返る。
  • Zeitwerk オートロード
    app/ 直下に置いたディレクトリは自動でオートロード対象になる。app/services の追加に設定変更は不要。
なぜこの設計か

三値を区別する理由：「見つからない」（nil）はユーザーの入力次第で普通に起こる正常系、「壊れている」（Error）はこちらの都合と無関係に起こる異常系。これを1つに潰すと、API 障害のときにも「見つかりませんでした」と嘘の案内をしてしまい、ユーザーは正しい ISBN を疑い始める。原因が違うものには違う出口を用意する。

タイムアウト5秒：Net::HTTP の既定は open/read 各60秒。設定しないと、Google が沈黙した場合にリクエストが Puma のスレッドを最大60秒占有し、外部 API の障害がアプリ全体の障害に転移する。外部依存には必ず時間の上限を切る。

例外の集約：ネットワーク系の例外は Timeout::Error / SocketError / SystemCallError…と多岐にわたる。呼び出し側にこの一覧を知らせるのは漏れのもとなので、境界（このクラス）で1つの GoogleBooks::Error に翻訳する。rescue 漏れ＝500 画面、を構造的に防ぐ。

ISBN 正規化を公開メソッドにした理由：問い合わせと @book.isbn への代入の両方で同じ正規化が要る。private にすると呼び出し側が正規表現を重複実装し、いずれ2つの正規化がズレる。

サーバーサイド反映（books#new + GET フォーム）

変更ファイル app/controllers/books_controller.rb ・ app/views/books/new.html.erb

「自動取得」の実体はただの GET フォーム。送信すると lookup_isbn パラメータ付きで登録画面自身に戻ってきて、new アクションが API を呼び、@book の初期値に詰めて描画する。新しいルーティングも、Stimulus コントローラも、JSON エンドポイントも作っていない。

app/controllers/books_controller.rb（抜粋）ruby
def new
  @book = Book.new
  prefill_from_google_books if params[:lookup_isbn].present?
end

private

# 発展要件6: ISBN から Google Books API で書誌情報を取得し、フォーム初期値に反映する。
# コントローラは結果の「状態」（@lookup_result）だけを決め、表示文言は view に置く。
# 取得失敗は登録を妨げない（メッセージを出して手入力にフォールバック）
def prefill_from_google_books
  @book.isbn = GoogleBooks.normalize(params[:lookup_isbn])
  volume = GoogleBooks.lookup(params[:lookup_isbn])

  if volume
    @book.title = volume.title
    @book.publisher = volume.publisher
    @book.published_year = volume.published_year
    @book.new_author_names = volume.authors.join("、")
    @lookup_result = :found
  else
    @lookup_result = :not_found
  end
rescue GoogleBooks::Error => e
  Rails.logger.warn("Google Books lookup failed: #{e.message}")
  @lookup_result = :error
end
app/views/books/new.html.erb（抜粋）erb

活用した Rails の機能

  • form_with url: / method: :get
    モデルに紐づかない検索フォームの定番。パラメータが book[...] にネストされず素の lookup_isbn で届く。
  • フォームビルダーの初期値解決
    form.text_field :title は @book.title を初期値に使う。コントローラで詰めるだけでビュー側の変更が不要だった理由。
  • before_action :require_admin
    既存の認可がそのまま効く。new の中で呼ぶ設計にしたので、lookup だけ認可を掛け忘れる余地がない。
  • Rails.logger.warn
    API 障害は握り潰さずログに残す。ユーザーには優しく、運用者には正確に。
なぜサーバーサイド反映か（Stimulus + JSON 案との比較）

部品が最少：Stimulus 案は「JSON エンドポイント＋Stimulus コントローラ＋fetch のエラーハンドリング＋JS 有効なシステムテスト」が必要になる。GET フォーム案は既存の new アクション1本で済み、通常の統合テストだけで HTML の value 属性まで検証できる。

状態と文言の分離：コントローラは @lookup_result（:found / :not_found / :error）という状態だけを決め、ユーザー向けの文言は view が持つ。文言をコントローラに書くと、コピー修正のたびにロジック層を触ることになり、テストも文字列比較に引きずられる。

そうしないとどうなるか：もし取得失敗を例外のまま上げると 500 画面で登録作業ごと中断する。もし rescue => e で何でも握り潰すと、コードのバグ（NoMethodError など）まで「取得に失敗しました」に化けて発見が遅れる。GoogleBooks::Error だけを rescue しているのはそのため。

新規ルーティングを作らなかった理由：new は既に「登録フォームを描く」責務を持つ。「初期値をどう用意するか」はその内側の変種であり、別アクションに切ると認可・レイアウト・エラーハンドリングを二重管理することになる。

03

複数著者への対応（new_author_names）

変更ファイル app/models/book.rb（仮想属性のリネームと拡張）・フォーム・seeds・既存テスト追随

Google Books は著者を authors: ["Dustin Boswell", "Trevor Foucher"] という配列で返す。ところが従来の仮想属性 new_author_name は1人分しか受け取れなかった。そこで new_tag_names（発展要件3）と同じ読点・カンマ区切りの複数対応へ拡張し、名前も複数形にリネームした。

app/models/book.rb（仮想属性と取り込み）ruby
attr_reader :new_author_names, :new_tag_names, :initial_stock_count
before_save :attach_new_authors

def new_author_names=(value)
  @new_author_names = value.to_s.strip.presence
end

# タグと同じく読点・カンマ区切りで複数登録できるようにする
# （Google Books API が著者を配列で返すため、複数著者を個別レコードとして扱う）
def attach_new_authors
  new_author_names.to_s.split(/[、,]/).map(&:strip).reject(&:blank?).uniq.each do |name|
    author = Author.find_or_create_by!(name: name)
    authors << author unless authors.exists?(author.id)
  end
end

コントローラ側は volume.authors.join("、") で配列を1つの入力欄用の文字列に畳み、保存時に attach_new_authors が再び個別の Author レコードへ分解する。画面上は1つのテキスト欄のままなので、手入力ユーザーの操作感は変わらない。

なぜリネームまでしたか（そうしないとどうなるか）

配列を単数フィールドに join して流し込むだけでも「動く」。しかしその場合 Author.find_or_create_by!(name: "Dustin Boswell、Trevor Foucher") が走り、2人の共著者が「読点入りの名前を持つ架空の1人」として DB に入る。著者名検索（必須要件5）はこの本を「Boswell」でヒットさせられなくなり、書籍↔著者の多対多（必須要件3）も事実上壊れる。外部 API の形状と自分のデータモデルの不一致は、境界でちゃんと変換するしかない。

名前を new_author_names（複数形）にしたのは、単数形のまま複数を受けると名前が仕様を偽るため。仮想属性は DB カラムではないのでマイグレーション不要、影響はフォーム・strong parameters・seeds・テストの呼び出し箇所のみで、すべて同一 PR 内で追随させた。

区切り文字を「、」と「,」の両対応にしたのは new_tag_names と同じ理由（日本語入力では読点が自然）。既存機能と同じルールに揃えることで、利用者が覚えることを増やさない。

テストの見どころ
test/models/book_test.rb に「読点・カンマ混在で3人登録できる」テストを追加（"Dustin Boswell、Trevor Foucher, 村上春樹" → Author.count が +3）。既存の5テストは属性名を追随させただけで、単一著者の挙動（strip・既存著者の再利用・重複紐付け防止）が後方互換で保たれていることの証明として残っている。

テストの全体マップ

層件数何を保証するか
サービス（google_books_test.rb）9ヒット／正規化／日付形式差（"2012-06" と "1914"）／任意項目の欠損／ヒットなし／空 ISBN は問い合わせない／500／タイムアウト／壊れた JSON

なぜこの構成か

スタブは HTTP 層に置く：GoogleBooks.lookup をメソッドごとモック化する手もあるが、それだと URL の組み立て・タイムアウト・JSON 解析という「本当に壊れやすい部分」が一度も実行されない。WebMock は Net::HTTP のソケット層で横取りするので、自分のコードは1行残らず本物が動く。

disable_net_connect! がないとどうなるか：スタブし忘れたテストが本物の Google に届き、「手元では通るが CI では落ちる」「Google のレート制限でランダムに落ちる」flaky テストが生まれる。実際、手動検証では本物の API が 429 を返した（→ 05 Q1）。テストがネットワーク非依存であることの価値をその場で実感することになった。

TDD の Red 確認：CLAUDE.md のルールどおり、サービス9件は「uninitialized constant GoogleBooks」、コントローラ・モデル分は「undefined method 'new_author_names='」と、意図した理由で失敗することを実装前に目視確認している。実装がなくても通るテスト（偽陽性）を仕込んでいない証拠になる。

想定問答 — 強みと、正直な弱点

レビュー会で聞かれそうな質問と回答の準備。強みは胸を張って、未実装は正直に + 「やるならこうする」をセットで

まず押さえる3つの強み

  • 三値の戻り値設計「見つからない」(nil) と「壊れている」(Error) を区別し、ユーザーへの案内が原因に対して正確（→ 01）
  • 実ネットワーク遮断のテストdisable_net_connect! でスタブ必須を仕組み化。500・タイムアウト・壊れた JSON まで機械的に再現（→ 04）
  • 依存ゼロ・JS ゼロランタイム gem 追加なし（Net::HTTP）・Stimulus なし（GET フォーム）。部品が少ないほど壊れる場所も少ない（→ 01, 02）

想定問答（クリックで回答を開く）

Q1Google Books API が落ちていたら登録作業はどうなりますか？

強み登録は止まりません。GoogleBooks::Error をコントローラで rescue して「取得に失敗しました。時間をおいて再試行するか、手入力で登録してください」と案内し、入力済みの ISBN はフォームに保持されます。実は手動検証の最中に本物の API が HTTP 429（レート制限）を返す場面に遭遇し、このフォールバックが設計どおり動くことを実機で確認済みです。外部 API は「いつでも落ちるもの」として設計しました。

Q2レート制限（429）に恒常的に当たるようになったら？

未実装キャッシュ・API キー・リトライは未実装です。現状は毎回素の匿名リクエストを投げるため、Google の IP 単位クォータに依存しています。やるなら：① Rails.cache.fetch(["google_books", isbn], expires_in: 1.week) で同一 ISBN の再問い合わせを止める（書誌情報はほぼ不変なのでキャッシュ向き。Solid Cache が導入済みなので追加部品も不要）、② Google Cloud Console で API キーを発行し credentials で管理、③ 429 のときだけ指数バックオフで1回リトライ。この順で効果が大きいです。

Q3なぜ Stimulus で画面遷移なしに反映しなかったのですか？

説明できる選択着手前に「Stimulus + JSON エンドポイント」案と比較して決めました。サーバーサイド案は JSON API・Stimulus コントローラ・fetch のエラー処理・JS 有効なシステムテストが全部不要になり、通常の統合テストで HTML の value 属性まで検証できるのが決め手です。登録前の新規フォームなので「再描画で入力が消える」デメリットも実質ありません。UX 要求が上がったら、new の検索処理を JSON でも返すようにして Stimulus を足す移行パスも見えています。

Q4編集画面でも ISBN から再取得できますか？

未実装登録（new）画面だけです。編集は「既に人間が確認したデータ」を外部データで上書きする操作になるため、意図しない上書き事故を避けて対象外にしました。やるなら：検索フォームと結果メッセージをパーシャルに切り出して edit にも置き、edit アクションに同じ prefill_from_google_books を通します。ただし「どの項目を上書きするか」のチェックボックスを付けるなど、上書きの合意を取る UI が先だと考えています。

Q5同じ ISBN で複数の候補が返ってきたらどうなりますか？

割り切りitems[0]（先頭の1件）だけを使います。ISBN は理論上1冊を特定するコードなので複数候補は例外的ですが、Google Books は版違いなどで複数返すことがあります。やるなら：parse を複数 Volume 返却に変え、候補をラジオボタンで一覧表示して管理者に選ばせます。今回は「自動取得後に人間が確認・修正してから保存する」フローなので、先頭1件＋手修正で実用上足りると判断しました。

Q6表紙画像や書籍説明は取得しないのですか？

未実装Google Books のレスポンスには imageLinks（書影 URL）や description もありますが、books テーブルに対応カラムがないため取得対象外にしました。要件の書籍属性（タイトル・ISBN・出版年・出版社・著者）に合わせています。やるなら：Volume に項目を足し、書影は URL をそのまま保存するか（手軽・ただし外部リンク切れリスク）、Active Storage でダウンロード保存するか（堅牢・ストレージ費用）の選択から入ります。

Q7API 呼び出し中、ユーザーの画面は固まりませんか？

説明できる選択同期呼び出しですが、open/read 各5秒のタイムアウトを設定してあり、最悪でも約10秒で必ず応答が返ります（通常は1秒未満）。限界も把握しています：待っている間は Puma のスレッドを1本占有するので、利用者が多いサービスなら Turbo Frame の遅延読み込みや Active Job 化で非同期にすべきです。今回は「管理者だけが使う登録補助機能」で同時実行がほぼないため、複雑さを増やさない同期型を選びました。

Q8ISBN 自体の形式チェック（桁数・チェックディジット）はありますか？

改善余地ハイフン・空白の正規化のみで、「13桁か」「チェックディジットが正しいか」の検証はありません。誤った ISBN は「見つかりませんでした」という API の結果として間接的に伝わるだけです。やるなら：Book に validates :isbn, format: { with: /\A\d{10}(\d{3})?\z/ } を足し、さらにチェックディジット計算をカスタムバリデータにします。既存データに旧形式（ハイフン付き）が残っているため、正規化の一括マイグレーションとセットで入れる必要がある、というところまで見えています。
