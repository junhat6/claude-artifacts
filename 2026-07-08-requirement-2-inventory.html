書誌とコピーの分離 — データ移行つきマイグレーション

発展要件 2（同じ本を複数冊登録できる）の土台。設計の選択肢は着手前に比較検討して決定

1本のマイグレーションで「テーブル新設 → 既存データの SQL 移行 → NOT NULL 化 → 外部キー付け替え → 部分ユニーク index の張り替え」まで行う。up / down を明示的に書き分けており、ロールバック可能。

db/migrate/20260706154432_introduce_book_copies.rb（up・抜粋）ruby
def up
  create_table :book_copies do |t|
    t.references :book, null: false, foreign_key: true
    t.timestamps
  end

  add_reference :rentals, :book_copy, foreign_key: true  # まず NULL 許可で追加

  # 既存データ移行: 各書籍に1冊のコピーを作り、既存の貸出をそのコピーへ付け替える
  execute <<~SQL
    INSERT INTO book_copies (book_id, created_at, updated_at)
    SELECT id, CURRENT_TIMESTAMP, CURRENT_TIMESTAMP FROM books
  SQL
  execute <<~SQL
    UPDATE rentals
    SET book_copy_id = (
      SELECT book_copies.id FROM book_copies
      WHERE book_copies.book_id = rentals.book_id
    )
  SQL

  change_column_null :rentals, :book_copy_id, false      # 全行埋まってから NOT NULL 化

  remove_index :rentals, name: "index_rentals_on_book_id_when_active"
  remove_reference :rentals, :book, foreign_key: true

  # 「同じコピーのアクティブな貸出は同時に1件まで」を DB 層で保証する部分ユニーク index
  add_index :rentals, :book_copy_id,
            unique: true,
            where: "returned_at IS NULL",
            name: "index_rentals_on_book_copy_id_when_active"
end

活用した Rails の機能

  • t.references + foreign_key
    null: false と FK 制約を DB 層に。宙に浮いたコピー行を構造的に禁止する。
  • execute（SQL 直書き）
    データ移行はモデルクラスに依存させない。マイグレーション実行時点のモデルは未来のコードなので、当てにすると壊れる。
  • change_column_null
    「NULL 許可で追加 → データを埋める → NOT NULL 化」の3段階。最初から NOT NULL だと既存行で即死する。
  • 部分ユニーク index（where:）
    貸出中（returned_at IS NULL）の行だけに一意制約。履歴の蓄積と排他制御を両立する要。
  • def up / def down
    データ移行を含むため change では自動逆転できない。down では book_id を復元してから book_copies を落とす。
なぜ「在庫数カラム」ではなく「コピーのテーブル分離」か

比較したのは A. books に stock_count カラムを足す案と B. book_copies テーブルを新設する案（採用）。

A だと「アクティブ貸出数 < stock_count なら貸出可」という集計比較になり、既存の部分ユニーク index では守れなくなる。同時アクセス対策は with_lock（悲観的ロック）へ切り替えが必要で、DB 層の最終防衛線を失う。さらに在庫数カラムと実際の貸出数という二重管理が生まれ、「貸出中が3件あるのに在庫数を2に減らす」といった不整合の入り口になる。

B なら「コピー1つは同時に1人まで」という行単位の一意制約に帰着するので、必須要件で作った三層防御（index＋バリデーション＋rescue_from）の対象を張り替えるだけで済む。在庫数は行数の導出値なので不整合が構造的に起きない。図書館の現実（1冊ずつが物理的に存在する）ともモデルが一致する。

02

在庫ベースの貸出 — 三層防御の張り替え

発展要件 2（在庫数を管理し、同時に複数人が借りられる）— この PR の技術的ハイライト

「借りる」を押すと空いているコピーが自動で割り当てられる。ユーザーはコピーを選ばない。必須要件で作った三層防御は、対象を book_id から book_copy_id に張り替えてそのまま機能し続ける。

第1層モデルのバリデーション役割：UX（丁寧なエラー文）

validate :copy_must_be_available, on: :create — 割り当てられたコピーが貸出中なら「は貸出中のため借りられません」で弾く。

空きコピーの SELECT と INSERT の間には隙間がある（check-then-act）。同時に2人が同じ空きコピーを掴み得る
第2層DB の部分ユニーク index役割：整合性の最終防衛線

returned_at IS NULL の行に限って book_copy_id を一意にする。後着の INSERT は物理的に失敗する。

制約違反は ActiveRecord::RecordNotUnique 例外として上がってくる
第3層コントローラの rescue_from役割：例外を UX に変換

「ちょうど最後の在庫が貸し出されたため借りられませんでした。」というフラッシュに変換して書籍詳細へ戻す。

app/models/book_copy.rb（available スコープ）ruby
class BookCopy < ApplicationRecord
  belongs_to :book
  has_many :rentals

  # 「アクティブな貸出を持たないコピー」= 空きコピー。サブクエリで NOT IN 相当を組む
  scope :available, -> { where.not(id: Rental.active.select(:book_copy_id)) }

  # rentals を preload しておけば追加クエリなしで判定できる
  def available?
    rentals.none? { |rental| rental.active? }
  end
end
app/models/book.rb（在庫の導出メソッド）ruby
has_many :copies, class_name: "BookCopy", dependent: :destroy
has_many :rentals, through: :copies   # 本の貸出履歴はコピー越しに集約

def stock_count
  copies.size
end

# copies: :rentals を preload しておけば追加クエリなしで数えられる
def available_stock_count
  copies.count { |copy| copy.available? }
end

def available_copy
  copies.available.first   # 貸出処理用。こちらは DB に聞く
end

def active_rental_for(user)
  return nil if user.nil?

  rentals.active.find_by(user: user)
end
app/controllers/rentals_controller.rb（空きコピーの自動割当）ruby
rescue_from ActiveRecord::RecordNotUnique, with: :copy_just_rented

def create
  copy = @book.available_copy
  if copy.nil?
    return redirect_to @book, alert: "在庫がないため借りられません。"
  end

  @rental = Rental.new(user: Current.user, book_copy: copy)
  if @rental.save
    redirect_to @book, notice: "「#{@book.title}」を借りました。"
  else
    redirect_to @book, alert: @rental.errors.full_messages.to_sentence
  end
end

private

def copy_just_rented
  redirect_to @book, alert: "ちょうど最後の在庫が貸し出されたため借りられませんでした。"
end

活用した Rails の機能

  • scope + サブクエリ
    where.not(id: Rental.active.select(:book_copy_id)) は SQL を組むだけで実行しない（遅延評価）。1クエリの NOT IN に合成される。
  • has_many :through（集約方向）
    著者の多対多で使った仕組みを「Book → copies → rentals」の集約に応用。book.rentals の書き味を維持。
  • size と count の使い分け
    size はロード済みなら SQL を打たない。copies.count { ... } はブロック付きなのでメモリ内集計。preload とセットで N+1 を回避。
  • includes のネスト
    一覧・詳細で includes(:authors, copies: :rentals)。書籍150冊でもクエリ数は一定。
なぜこの設計か（そうしないとどうなるか）

「空きコピーの判定」を2系統用意しているのが読みどころ。表示系（available_stock_count）は preload 済みの rentals を Ruby 側で数え、一覧で150冊表示しても追加クエリゼロ。貸出系（available_copy）はサブクエリで DB に聞き、その時点の最新状態で1件取る。1系統に統一すると、表示は N+1 になるか、貸出は古いキャッシュを掴むかのどちらかを引き受けることになる。

張り替えの意味：必須要件の三層防御は「本は同時に1人」を守っていた。在庫対応で本単位の一意制約は成立しなくなる（複数人が借りられるのが要件）が、「コピーは同時に1人」は依然として行単位の一意制約で表現できる。制約の粒度を1段下げるだけで、DB 層の最終防衛線ごと新要件に持ち越せた。これが 01 で B 案を選んだ最大の配当。

テストの見どころ
test/controllers/rentals_controller_test.rb の「他のユーザーが貸出中でも在庫が残っていれば借りられる」が要件の核心（コピーAが貸出中の本を別ユーザーが借り、コピーBが割り当たることまで検証）。test/models/rental_test.rb では従来どおり save!(validate: false) で第1層を迂回し、DB 層が単体で二重貸出を弾くことも検証している。

03

二重貸出の禁止 — 本単位の制約はアプリ層で

要件外だが実務上必須と判断した追加仕様（設計判断として PR に明記）

在庫が2冊あると「同じ人が同じ本を2冊借りる」ことが可能になってしまう。これを Rental のバリデーションで禁止した。コピー単位の制約（→ 02）と違い、この制約は DB の index では表現できない点が設計の見どころ。

app/models/rental.rb（全文）ruby
class Rental < ApplicationRecord
  belongs_to :user
  belongs_to :book_copy
  has_one :book, through: :book_copy   # rental.book の書き味を維持

  scope :active, -> { where(returned_at: nil) }

  validate :copy_must_be_available, on: :create
  validate :must_not_rent_same_book_twice, on: :create

  def active?
    returned_at.nil?
  end

  private

  def copy_must_be_available
    return if book_copy.blank?
    return unless Rental.active.exists?(book_copy_id: book_copy_id)

    errors.add(:book_copy, "は貸出中のため借りられません")
  end

  def must_not_rent_same_book_twice
    return if user.blank? || book_copy.blank?
    return unless user.rentals.active.joins(:book_copy).exists?(book_copies: { book_id: book_copy.book_id })

    errors.add(:base, "すでにこの本を借りています")
  end
end

活用した Rails の機能

  • has_one :through
    Rental → BookCopy → Book と1段挟んでも rental.book で書誌に届く。返却処理のコードは無修正で動いた。
  • joins + exists?
    「このユーザーのアクティブ貸出のうち、同じ book_id のコピーを指すものがあるか」を1クエリで判定。SELECT 1 ... LIMIT 1 に落ちる。
  • errors.add(:base, ...)
    特定の属性の問題ではなくレコード全体の状態異常なので :base に付ける。フラッシュにそのまま出せる文にする。
  • ガード節による早期 return
    belongs_to の presence 検証と重複したエラーを出さないよう、対象が blank なら黙って抜ける（既存バリデーションと同じ流儀）。
なぜアプリ層だけなのか（そうしないとどうなるか）

「コピーは同時に1人」は rentals の行の中に book_copy_id があるから index で守れる。しかし「同じ本を二重に借りない」の判定には book_copies を JOIN しないと book_id が分からず、単一テーブルへの index では表現できない。DB 層で守るならトリガーか排他ロックが要るが、この制約は破られても「同じ本のコピーを2冊持つ人が現れる」だけでデータ破壊がない。整合性リスクと実装コストが釣り合わないので、アプリ層のみと判断した。

逆にこのバリデーションを入れないと、詳細画面で「借りる」を2回押した人が在庫を2冊とも占有してしまい、「複数人が借りられる」という要件の趣旨が実質的に損なわれる。

04

在庫の登録・追加・削除

発展要件 2（同じ本を複数冊登録できる）の操作系。履歴保護のガードつき・操作は管理者のみ（発展要件1と統合済み）

登録フォームでは冊数を仮想属性で受けて保存後にコピーを一括作成。登録後は書籍詳細の在庫管理テーブルから1冊ずつ追加・削除できる（管理者のみ）。貸出履歴のあるコピー（と本）は削除できない。

冊数の仮想属性 — new_author_name と同じパターン

app/models/book.rb（initial_stock_count まわり）ruby
attr_reader :new_author_name, :initial_stock_count

validates :initial_stock_count, numericality: { only_integer: true, greater_than_or_equal_to: 1 },
          allow_nil: true, on: :create

after_create :create_initial_copies

def initial_stock_count=(value)
  @initial_stock_count = value.presence&.to_i   # 空文字は nil（→ 既定1冊）、"abc" は 0（→ 検証で弾く）
end

private

def create_initial_copies
  (initial_stock_count || 1).times { copies.create! }
end

削除ガード — コールバックの実行順序が肝

app/models/book.rb ・ book_copy.rb（削除ガード）ruby
class Book < ApplicationRecord
  has_many :copies, class_name: "BookCopy", dependent: :destroy
  # prepend しないと copies の dependent: :destroy が先に走り、
  # 履歴チェックの前にコピー削除が始まってしまう
  before_destroy :must_not_have_rental_history, prepend: true
end

class BookCopy < ApplicationRecord
  # 貸出履歴は「誰が何を借りたか」の記録なので、コピーを消して履歴を失わないよう削除を止める
  before_destroy :must_not_have_rental_history

  def must_not_have_rental_history
    return unless rentals.exists?

    errors.add(:base, "貸出履歴があるため削除できません")
    throw :abort
  end
end
app/controllers/book_copies_controller.rb（新設・抜粋）＋ config/routes.rbruby
class BookCopiesController < ApplicationController
  before_action :require_admin   # 在庫の増減は管理者のみ（発展要件1の Authorization concern）
  before_action :set_book

  def create
    @book.copies.create!
    redirect_to @book, notice: "「#{@book.title}」の在庫を1冊追加しました。"
  end

  def destroy
    copy = @book.copies.find(params[:id])   # 別の本のコピー ID を渡されても 404
    if copy.destroy
      redirect_to @book, notice: "「#{@book.title}」の在庫を1冊削除しました。", status: :see_other
    else
      redirect_to @book, alert: copy.errors.full_messages.to_sentence, status: :see_other
    end
  end
end

# config/routes.rb
resources :books do
  resources :rentals, only: [ :create ]
  resources :copies, controller: "book_copies", only: [ :create, :destroy ]
end

活用した Rails の機能

  • 仮想属性（attr_reader + カスタム setter）
    DB カラムでないフォーム入力を受ける、このリポジトリの既存パターン（new_author_name）を踏襲。
  • before_destroy prepend: true
    関連の dependent: :destroy コールバックより前にガードを差し込む。コールバックは宣言順に積まれるため明示が必要。
  • throw :abort
    コールバックチェーンを止めて destroy を false 返しにする Rails 5+ の正式な中断方法。例外ではないので rescue 不要。
  • ネストリソース + controller:
    URL は /books/:book_id/copies、コントローラ名は BookCopiesController。ルート名と実装名を分離できる。
  • @book.copies.find(...)
    親スコープからの find が認可を兼ねる。返却処理（Current.user.rentals.active.find）と同じ定番イディオム。
なぜ restrict_with_error ではなく before_destroy か

最初は has_many :rentals, dependent: :restrict_with_error で実装した。テストは通ったが、実サーバで画面フローを検証したところフラッシュに「Cannot delete record because dependent rentals exist」と英語のデフォルトメッセージが出た（このアプリは i18n のロケールを英語のまま、ユーザー向け文言だけ日本語で書く方針のため）。テストを日本語メッセージ期待に書き換えて Red を確認してから、Book 側と同じ before_destroy + throw :abort パターンに統一した。機能・UX・パターンの一貫性が揃い、「テストが通る」と「ユーザーに正しく見える」は別物だと確認できた例。

そもそも削除を止めるのは、rentals が「誰が何を借りたか」の監査記録だから。dependent: :destroy にすると本を消した瞬間に貸出履歴ごと消え、nullify だと宙に浮いた履歴が残る。どちらも履歴要件（発展要件 4）と両立しない。

発展要件1（管理者権限）との統合
発展要件1（PR #10）が先にマージされたため main を取り込んで統合した。BookCopiesController に require_admin を適用し、詳細画面の在庫管理テーブルを admin? で出し分け。統合時も「一般ユーザーは在庫を追加できない」テストを先に書いて Red（一般ユーザーでも追加できてしまう）を確認してから before_action を足す TDD の手順を踏んだ。在庫数の表示は貸出判断の材料なので全ユーザーに見せたまま。

想定問答 — 強みと、正直な弱点

レビュー会で聞かれそうな質問と回答の準備。強みは胸を張って、弱点は正直に + 「やるならこうする」をセットで

まず押さえる3つの強み

  • 三層防御の「張り替え」部分ユニーク index の対象を book_copy_id に変えるだけで、DB 層の最終防衛線ごと在庫要件に持ち越した（→ 02）
  • 在庫数は導出値在庫数カラムを持たずコピーの行数から導出。カウンタと実態のズレが構造的に起きない（→ 01）
  • 実データ移行つき可逆マイグレーション既存の書籍・貸出を SQL で移行し、down も実装。本番の Fly.io にそのまま適用できる（→ 01）

想定問答（クリックで回答を開く）

Q1最後の1冊を同時に2人が借りたらどうなりますか？

強み片方だけが借りられます。両者の available_copy が同じ空きコピーを返し得ますが、部分ユニーク index が後着の INSERT を弾き、rescue_from が「ちょうど最後の在庫が貸し出されたため借りられませんでした。」に変換します。実サーバで同一フォームからの二重 POST（2タブ相当）を再現し、この動作を確認済みです。

正直な補足として、在庫が2冊以上空いている場合も両者が同じ「最初の空きコピー」を掴んで片方が弾かれることがあり得ます（残りがあるのにエラーになる）。再度押せば成功しますが、やるなら rescue_from 内で別の空きコピーを取り直して1回リトライする実装が考えられます。

Q2books に在庫数カラムを足す方が簡単では？なぜテーブルを分けたのですか？

強み在庫数カラム方式だと貸出可否が「アクティブ貸出数 < 在庫数」という集計比較になり、行単位のユニーク制約では守れません。同時アクセス対策は悲観的ロック（with_lock）頼みになり、DB 層の最終防衛線を失います。さらに「貸出中3件のまま在庫数を2に減らす」といった不整合の入り口になる二重管理も生まれます。テーブル分離なら三層防御が張り替えだけで維持でき、在庫数も導出値なので破綻しない — 変更量は増えるが構造的に安全な方を選びました（着手前に両案を比較して決定）。

Q3同じユーザーが2タブで同時に「借りる」を押したら、同じ本を2冊持てませんか？

弱点理論上あり得ます。「同じ本を二重に借りない」制約は book_copies を JOIN しないと判定できず、単一テーブルへの index では表現できないため、アプリ層のバリデーションのみです。すり抜けても「同じ本のコピーを2冊持つ人が現れる」だけでデータ破壊はなく、コピー単位の整合性は index が守り続けます。やるなら：貸出作成を book.with_lock で本単位に直列化するか、SQLite なら書き込みが単一ライターである性質も踏まえて許容する、という判断まで含めて説明します。

Q4一覧で全書籍の在庫数を出すと N+1 になりませんか？

強みincludes(:authors, copies: :rentals) で preload し、available_stock_count は copies.count { |copy| copy.available? } というブロック付き count（＝メモリ内集計）なので追加クエリは出ません。size はロード済み関連に SQL を打たない点も使い分けています。preload を忘れた場合は開発中の bullet gem が検知します。一方、貸出処理の available_copy はあえて DB に聞く（サブクエリ）ことで最新状態を取る — 表示系と処理系で鮮度と効率のトレードオフを分けています。

Q5在庫の追加・削除は誰でもできてしまうのですか？

強み管理者のみです。発展要件 1（PR #10）が先にマージされたため main を取り込み、BookCopiesController に require_admin（Authorization concern）を適用、詳細画面の在庫管理テーブルも admin? で出し分けました。一般ユーザーが URL 直叩きで POST しても root へリダイレクトされます（コントローラテストで検証済み）。一方、在庫数の表示（全◯冊中◯冊が貸出可能）は全ユーザーに見せたままにしています — 借りられるかどうかの判断材料であり、隠す理由がないためです。統合時も「一般ユーザーは在庫を追加できない」テストを先に書いて Red を確認してから before_action を足す、という TDD の手順を踏みました。

Q6既に登録済みの本や貸出中のデータはどうなりましたか？

強みマイグレーション内で「各書籍にコピーを1冊ずつ作成 → 既存の貸出をそのコピーへ UPDATE → NOT NULL 化 → 旧 book_id カラム削除」を SQL で行い、既存データを無傷で移行しました。旧制約では同一書籍のアクティブ貸出は最大1件だったので、1冊への集約で新しい部分ユニーク index に違反することもありません。down も書いてあり、ロールバックすると book_id が復元されます。

Q7返却済みのコピーがまた借りられるのはどういう仕組みですか？

強みユニーク index が「部分」だからです。制約対象は WHERE returned_at IS NULL の行だけなので、返却（returned_at に時刻が入る）と同時にそのコピーは制約の対象外になり、次の貸出 INSERT が通ります。available スコープも Rental.active（returned_at が NULL）のサブクエリなので、返却の瞬間に空き在庫としてカウントされます。テストにも「返却済みになったコピーは再度借りられる」があります。
