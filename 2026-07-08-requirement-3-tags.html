発展要件 3（タグ・ジャンル管理。書籍（多）― タグ（多）の多対多）に対応

追加したテーブルは tags と book_tags の2つだけ。「ジャンル（技術書、小説…）」と「タグ（Ruby、入門…）」を別テーブルに分けず、単一の Tag モデルとして扱う設計を選んだ（着手前にユーザーと合意）。books ↔ authors の多対多と完全に相似形になっている。

01

多対多の土台 — マイグレーションと3モデル

tags / book_tags テーブル、Tag / BookTag モデル、Book への関連追加

マイグレーション1本で2テーブルを作る。NOT NULL・外部キー・ユニーク index という DB 層の制約をこの時点で全部入れるのがこのリポジトリの流儀（在庫管理・貸出と同じ）。

db/migrate/20260707090000_create_tags_and_book_tags.rb（全文）ruby
class CreateTagsAndBookTags < ActiveRecord::Migration[8.1]
  def change
    create_table :tags do |t|
      t.string :name, null: false

      t.timestamps
    end
    add_index :tags, :name, unique: true

    create_table :book_tags do |t|
      t.references :book, null: false, foreign_key: true
      t.references :tag, null: false, foreign_key: true

      t.timestamps
    end
    add_index :book_tags, [ :book_id, :tag_id ], unique: true
  end
end
app/models/tag.rb ・ book_tag.rb（スキーマ注釈を除く全文）ruby
class Tag < ApplicationRecord
  has_many :book_tags, dependent: :restrict_with_error  # 使用中タグは消せない
  has_many :books, through: :book_tags

  validates :name, presence: true, uniqueness: true
end

class BookTag < ApplicationRecord
  belongs_to :book
  belongs_to :tag

  validates :tag_id, uniqueness: { scope: :book_id }  # アプリ層の重複防止
end
app/models/book.rb（関連の追加部分）ruby
has_many :book_tags, dependent: :destroy  # 本を消せばタグ付けも道連れ
has_many :tags, through: :book_tags

活用した Rails の機能

  • has_many :through
    中間モデルを明示する多対多。habtm と違い、将来「タグ付け日時」「付けた人」等の属性を中間に足せる。
  • t.references ... foreign_key: true
    FK カラム＋外部キー制約＋単独 index を1行で。孤児レコード（存在しない本へのタグ付け）を DB が拒否。
  • add_index ..., unique: true
    tags.name の一意性と (book_id, tag_id) の複合一意性。並行リクエストでも重複を通さない最終防衛線。
  • dependent: の使い分け
    Book 側は :destroy、Tag 側は :restrict_with_error。同じ関連でも向きで削除ポリシーが変わる。
  • validates uniqueness: { scope: }
    複合ユニークのアプリ層版。ユーザーに見せられるエラーメッセージを担当。
  • annotaterb
    マイグレーション実行と同時にモデル・テストへスキーマ注釈を自動付与。コードだけでテーブル構造が読める。
なぜこの設計か

ユニーク制約は二層で張る。アプリ層の uniqueness だけだと、同時リクエストが両方バリデーションを通過して重複が入り得る（SELECT と INSERT の間に隙間がある）。逆に DB 層だけだと、普通の重複入力までもが例外＝500 エラー画面になる。モデル層＝UX、DB 層＝整合性という役割分担は、貸出の三層防御（必須要件4）から一貫している。

dependent の非対称も Author とまったく同じ理屈。本を消すときタグ付けの行だけ残っても無意味なので Book 側は :destroy。一方 Tag は複数の本から共有されるので、使用中に消すと他の本のタグまで消える。専用削除画面が無い現状では :restrict_with_error で誤削除から守る保守的な既定にしておく。

habtm を選ばないのは必須要件3の ADR と同じ判断。中間テーブルにモデルが無いと、属性を足したくなった時点でテーブル再設計になる。

02

タグ付与フォーム — new_tag_names 仮想属性

書籍フォームで「既存タグの複数選択」と「新規タグの一括作成」を両対応

著者で使った仮想属性パターン（new_author_name）を拡張し、「、」「,」区切りで複数タグを一度に作成・紐付けできる new_tag_names を追加した。DB カラムではない入力を attr_reader ＋カスタム setter で受け、before_save で取り込む。

app/models/book.rb（仮想属性と取り込み）ruby
attr_reader :new_author_name, :new_tag_names, :initial_stock_count
before_save :attach_new_tags

def new_tag_names=(value)
  @new_tag_names = value.to_s.strip.presence  # 空白だけの入力は nil に潰す
end

private

def attach_new_tags
  # 読点・カンマのどちらでも区切れるようにする（日本語入力での「、」を想定）
  new_tag_names.to_s.split(/[、,]/).map(&:strip).reject(&:blank?).uniq.each do |name|
    tag = Tag.find_or_create_by!(name: name)
    tags << tag unless tags.exists?(tag.id)
  end
end
app/views/books/_form.html.erb（追加部分・抜粋）erb
<div class="space-y-2">
  <%= form.label :tag_ids, "タグ・ジャンル（既存）", class: "label" %>
  <%= form.collection_select :tag_ids, @tags, :id, :name, {},
        { multiple: true, class: "select select-bordered w-full min-h-32" } %>
</div>

<div>
  <%= form.label :new_tag_names, "新しいタグを追加", class: "label" %>
  <%= form.text_field :new_tag_names, class: "input input-bordered w-full",
        placeholder: "例: 技術書、Ruby（「、」区切りで複数登録）" %>
</div>
app/controllers/books_controller.rb（strong parameters）ruby
def book_params
  params.require(:book).permit(:title, :isbn, :published_year, :publisher, :new_author_name, :new_tag_names, :initial_stock_count, author_ids: [], tag_ids: [])
end

活用した Rails の機能

  • 仮想属性（attr_reader ＋カスタム setter）
    DB カラムに無いフォーム入力をモデルで受ける定番。setter で正規化（strip / presence）まで済ませる。
  • find_or_create_by!
    同名タグは再利用、無ければ作成。「技術書」タグが本の数だけ増殖するのを防ぐ。
  • before_save コールバック
    create / update 両方で発火するため、新規登録でも編集でも同じ経路でタグが取り込まれる。
  • collection_select multiple
    既存タグの複数選択。tag_ids= は has_many :through が自動生成する差し替え用 setter。
  • strong parameters の配列許可
    tag_ids: [] の明示が必要。書き忘れると Rails はエラーにせず黙って捨てる（気づきにくいバグの温床）。
なぜこの設計か

なぜ複数一括入力にしたか：著者は1冊に1〜2人だが、タグは「技術書・Ruby・入門」のようにまとめて付けるのが自然な単位。1個ずつしか足せないと、タグがまだ無い初期ほど保存を何往復もさせられる。区切り文字に「、」を含めたのは、日本語 IME で打ちやすい記号だから（カンマだけだと入力のたびに IME 切り替えが要る）。

取り込みロジックがタグ付与の重複に3段構えで耐える：入力内の重複は uniq が畳み、既存タグとの重複は find_or_create_by! が再利用し、紐付け済みタグは tags.exists? ガードがスキップする。これが無いと「既に付いているタグ名を入れて保存 → 複合ユニーク index 違反で 500」になる。

ロジックをモデルに置くのは、コントローラに書くと new / create / edit / update の4箇所で同じ処理が要るから。モデルの before_save なら保存経路が何であれ1箇所で済み、モデル単体テストで検証できる。

03

タグ絞り込み — with_tag スコープ

一覧・詳細のタグバッジをクリックすると、そのタグが付いた本だけに絞り込む

絞り込みロジックは検索（scope :search）と同じくモデルのスコープに置いた。blank なら all を返す設計にしてあるので、検索キーワードとタグ絞り込みをメソッドチェーンでそのまま合成できる。

app/models/book.rb（scope :with_tag）ruby
scope :with_tag, ->(tag_id) {
  return all if tag_id.blank?

  joins(:tags).where(tags: { id: tag_id })
}
app/controllers/books_controller.rb（index）ruby
def index
  # 存在しない tag_id は絞り込みなしとして扱う（フィルタ表示も出さない）
  @current_tag = Tag.find_by(id: params[:tag_id])
  @books = Book.search(params[:q]).with_tag(@current_tag&.id)
               .includes(:authors, :tags, copies: :rentals).order(created_at: :desc)
end
app/views/books/index.html.erb（タグバッジと絞り込み表示・抜粋）erb
<%# 検索フォームに hidden で tag_id を持たせ、検索してもタグ絞り込みが外れないようにする %>
<%= form.hidden_field :tag_id, value: params[:tag_id] %>

<% if @current_tag %>
  <div class="mb-4 flex items-center gap-2">
    <span class="text-sm text-base-content/60">タグで絞り込み中:</span>
    <span class="badge badge-primary"><%= @current_tag.name %></span>
    <%= link_to "解除", books_path(q: params[:q].presence), class: "link link-hover text-sm" %>
  </div>
<% end %>

<%# 一覧の各行：タグをクリック可能なバッジとして表示（現在の検索語 q は引き継ぐ） %>
<% book.tags.each do |tag| %>
  <%= link_to tag.name, books_path(tag_id: tag.id, q: params[:q].presence),
        class: "badge badge-outline badge-primary" %>
<% end %>

活用した Rails の機能

  • スコープの合成
    Book.search(q).with_tag(id).includes(...) と数珠つなぎにできる。各スコープが「blank なら all」を守っているのが合成可能性の鍵。
  • joins(:tags)
    INNER JOIN で「タグが付いている本」だけに絞る。単一 tag_id との一致なので行の複製が起きず distinct 不要。
  • find_by（例外を投げない検索）
    find だと不正な tag_id で 404。find_by は nil を返すので「絞り込みなし」に自然に落とせる。
  • includes(:tags)
    一覧で各行のタグを表示するため eager load。無いと書籍数ぶんのタグ取得クエリが走る（N+1）。
なぜこの設計か

スコープをモデルに置くのは検索実装（必須要件5）と同じ理由。コントローラに where を直書きすると、検索との組み合わせ分岐（q あり／tag あり／両方）がコントローラ内で爆発する。「blank なら all」のスコープ2つを常に両方通す形にすれば、分岐そのものが消える。

存在しない tag_id を「絞り込みなし」に落とすのは UX の選択。URL を手で書き換えた・削除済みタグの古いリンクを踏んだ、という場合に「0件です」と出すより、全件表示に戻して絞り込み表示も出さないほうが行き止まりにならない。@current_tag&.id（safe navigation）でスコープに nil が渡り、all に落ちる。

hidden_field で tag_id を運ぶのは、GET フォームは action の URL クエリを引き継がないから。これが無いと「タグで絞り込んでから検索」した瞬間に絞り込みが静かに外れる。逆方向（タグクリック時に q を渡す）も同様で、検索とタグ絞り込みが直交する2軸として共存する。

  • Active Record クエリインターフェイス — scope・joins・includes（N+1 対策）の使い分けがすべてこの章にある。

想定問答 — 強みと、正直な弱点

レビュー会で聞かれそうな質問と回答の準備。強みは胸を張って、未実装は正直に + 「やるならこうする」をセットで

まず押さえる3つの強み

  • パターンの一貫性join モデル明示・二層ユニーク制約・仮想属性・dependent の非対称——必須要件の Book–Author と同じ設計語彙で2本目の多対多を組んだ（→ 01, 02）
  • 合成可能なスコープ設計「blank なら all」を守る search × with_tag で、検索とタグ絞り込みが分岐ゼロで直交する。hidden_field で状態も相互保持（→ 03）
  • Red の理由まで見る TDD15件全部が「実装が無い」ことを理由に落ちるのを確認してから着手。偽陽性の混入をプロセスで防いだ（→ 04）

想定問答（クリックで回答を開く）

Q1「ジャンル」と「タグ」を区別しなくてよかったのですか？

説明できる選択要件は「ジャンルやタグをつけられる」「書籍―タグの多対多」で、両者の区別は求めていません。区別を持ち込むと Genre/Tag 分離ならテーブル4つ、kind カラム方式でもフォームと検証が2系統になり、得られるのは表示上の分類だけです。着手前に選択肢3案（単一 Tag／kind カラム／テーブル分離)を比較して単一 Tag を選びました。将来区別が必要になったら、tags に kind カラムを足す enum 方式へマイグレーション1本＋既定値で移行でき、単一テーブルにしておいたことが移行コストを下げる方向に働きます。

Q2タグの改名や削除はどこでやるのですか？

未実装タグ専用の管理画面（CRUD）は意図的に作っていません。Author に専用画面が無いのと同じ管理粒度に揃えています。書籍からタグを外すことはできますが、タグ自体の改名・削除はできません。

やるなら：管理者専用の TagsController（index / update / destroy）を追加します。認可は既存の require_admin を before_action で掛けるだけ、削除は dependent: :restrict_with_error が既に効いているので「使用中タグは消せません」のエラーメッセージ付きで安全に落ちます。土台の制約が先にあるので、画面は薄く作れます。

Q3どの本からも外された「未使用タグ」が溜まりませんか？

改善余地溜まります。書籍からタグを外しても tags の行は残り、現状それを掃除する仕組みはありません。ただし未使用タグは書籍フォームの選択肢に出続けるだけで、一覧・詳細の表示や絞り込み結果を壊すことはありません。

やるなら：2案あります。①BookTag の after_destroy で「最後の1件を外したときにタグも消す」——即時性はあるが、直後に再利用するタグまで消える。②未使用タグを Tag.where.missing(:book_tags)（Rails 7 の missing）で洗い出す定期ジョブか管理画面——運用で判断できる余地が残る。管理画面（Q2）とセットなら②が自然です。

Q4「技術書 かつ Ruby」のような複数タグの絞り込みはできますか？

未実装絞り込みは1タグずつです（バッジをクリックするとそのタグに切り替わる）。AND 検索は未対応です。

やるなら：with_tags(tag_ids) スコープを追加し、joins(:tags).where(tags: { id: tag_ids }).group(:id).having("COUNT(DISTINCT tags.id) = ?", tag_ids.size) で「指定タグを全部持つ本」を取ります。JOIN で1本=タグ数ぶんの行に増えることを利用して件数一致で判定する、多対多 AND 検索の定石です。UI 側は選択中タグの配列を URL クエリ（tag_ids[]）で持ち回る変更が必要で、モデルよりビューの改修が大きくなります。

Q5同時に2人が同じ名前の新規タグで保存したらどうなりますか？

正直に言うと隙間ありfind_or_create_by! は「SELECT → 無ければ INSERT」の2ステップなので、同時実行では両方が「無い」と判定し得ます。その場合、後着の INSERT は tags.name のユニーク index（DB 層の最終防衛線）に弾かれて例外になり、タグの重複行は入りませんが、ユーザーには 500 画面が見えます。貸出機能では同種の例外を rescue_from で UX に変換しましたが、タグでは未対応です。書籍登録は管理者のみ・同名タグの同時作成という条件の狭さから、発生確率を考えて割り切りました。

やるなら：find_or_create_by! を Tag.create_or_find_by!(name: name)（Rails 6+）に差し替えます。こちらは先に INSERT を試みてユニーク制約違反なら既存行を取り直すため、この競合をリトライなしで吸収できます。

Q6「Ruby」と「ruby」は別タグになりますか？（表記ゆれ）

改善余地別タグになります。tags.name の一意性は大文字小文字を区別したままで、正規化は前後空白の除去（strip）だけです。authors.name も同じ割り切りなので、リポジトリ内では一貫しています。

やるなら：表示名は入力どおり残したいので、単純な normalizes ... downcase は使えません。比較用の name_normalized カラム（downcase を保存、ここにユニーク index）を足し、find_or_create_by! の検索キーをそちらに切り替えます。「表示用の値と一意性判定用の値を分ける」のが表記ゆれ対策の定石です。

Q7タグ絞り込みで一覧が遅くなりませんか？（N+1）

強み一覧は includes(:authors, :tags, copies: :rentals) でタグも著者も在庫もまとめて eager load しており、書籍数が増えてもクエリ本数は一定です。絞り込み自体は joins による INNER JOIN で、book_tags には book_id / tag_id 両方の index（t.references が自動生成）が張ってあるので index が効きます。開発環境では bullet gem が N+1 を検知する体制もあります。残る課題は全件表示そのもの（ページネーション未導入、発展要件5）で、これはタグ機能ではなく一覧全体の課題として把握しています。
