IdeaFlow 技術選定の再評価 — 実装負荷を除外した場合

IdeaFlow — 技術選定メモ / 2026-07-07

実装負荷をゼロと仮定したとき、IdeaFlow のベスト構成はどこに動くか

既存の調査（docs/tech-stack-research.md）は「認証の再構築・テスト戦略の再構築・学習コスト」という移行負荷を重く見て、現行の Next.js + Supabase + Vercel 継続を推奨した。本メモはその前提を外し、同じ MVP 要件をコスト・ユーザビリティ・機能との構造適合・継続性の4軸だけで採点し直す。

結論 — VERDICT

1位 / BEST FIT

Cloudflare Workers + Durable Objects
（1ルーム = 1 権威サーバー）+ Clerk + React SPA

ルームという「小さな権威サーバー」をそのままインフラの単位にできる唯一の構成。ステルス投票は「サーバーが送らないものはクライアントに存在しない」という最強のランタイム保証になり、コストはアイドル無料・配信無料でこのアプリの利用パターン（バースト利用・長い待機）に完全に一致する。定常運用でも月 0〜5 ドル。

2位
Convex + Clerk — 「クライアントは DB に触れない」という構造で可視性制御をバイパス不能にできる。正しさの保証は最強クラスだが、高頻度なドラッグ配信が関数呼び出し課金と相性が悪く、実質 Pro（$25/月〜）前提。
3位
Next.js + Supabase + Vercel（現行） — RLS + pgTAP の宣言的な認可保証は唯一無二。ただし Free プランの「1週間放置で自動停止」と「メッセージ数課金 × ドラッグ配信」がこのアプリの使われ方と噛み合わない。実装負荷を評価に戻せば1位に返り咲く合理解であり、既存調査の結論とは矛盾しない。

01このアプリの正体 — 要件を4つに蒸留する

PRD（デザインスプリント型ブレスト支援）と PoC の結果から、技術選定を左右する要件は次の4つに集約される。

#要件技術的な意味
R1フェーズ状態機械①〜⑤のフェーズ・ステップを全員が同期して進む。遷移はホストのみ。再接続時に現在状態へ復帰。
R2付箋の行単位同期作成・編集・移動が全員に反映。競合は「最後勝ち」で許容。文字単位の共同編集（CRDT）は不要。
R3サーバー強制の可視性制御個人ワーク中は自分の付箋のみ、ステルス投票中は自分の票のみ。DevTools を開いても他人の票が見えてはならない。
R4プレゼンス + 高頻度エフェメラル在室表示、サーバー時刻基準のタイマー、ドラッグ中座標の約 80ms 間隔配信（PoC 実測 69〜99ms）。

加えて、選定の重みを決めるのは要件そのものより利用パターンだ。

  • 少人数・セッション型 — 1ルーム 3〜8人、1〜3時間で完結。同時接続のピークは低い。
  • 極端にバースト的 — ハッカソン週末に集中し、平日はほぼ無人。アイドル時間が総時間の 95% 以上を占める。
  • ユーザーは学生 — 収益化の予定がなく、運用費は「無料〜数ドル」が持続ラインになる。
  • 会場の Wi-Fi は不安定 — 切断・再接続からの復帰品質が、そのままユーザー体験になる。

02候補を最初に絞る一撃 — これは「共同編集ドキュメント」ではない

コラボアプリの技術選定では Yjs / Automerge / Liveblocks Storage といった CRDT・local-first 系がまず候補に挙がる。しかし IdeaFlow では R3 がこれらを構造的に除外する。

CRDT は「同じドキュメントを全参加者に複製し、どの順で編集が届いても収束させる」仕組みだ。つまり全員が全データを持つことが前提にある。ところがステルス投票と個人ワークは「人によって見えるものが違う」機能であり、票をドキュメントに入れた瞬間、暗号化でもしない限り DevTools から全票が読めてしまう。回避するには票と個人付箋を CRDT の外——つまり認可付きのサーバー——へ出すしかなく、そうするとアプリの核心機能がメインの同期基盤の外側で動く二重アーキテクチャになる。

したがって候補は「サーバーが見せるものを選べる」アーキテクチャに絞られる。強制のやり方は3通りある。

強制モデル仕組み代表保証の性質
宣言（pull 型） クライアントが DB に問い合わせ、行レベルのポリシーが読める行を絞る Supabase RLS ポリシー = 仕様書。pgTAP で回帰テスト可能。宣言の書き漏れが唯一の敵。
選択的送信（push 型） 権威サーバーが各クライアントへ「見せてよいものだけ」送る Durable Objects 送らないものはクライアントに存在しない。ランタイム保証は最強だが、「送らないこと」の保証は手続き的コードとテストに依る。
構造（関数経由） クライアントは DB に直接触れず、全読み取りがサーバー関数を通る Convex バイパス経路が存在しない。フィルタは関数内のコードなので、こちらもテストで担保。

03候補の評価

実装・移行の負荷は採点しない。評価軸はコスト / ユーザビリティ / R3 との構造適合 / 継続性リスクの4つ。

B Cloudflare Workers + Durable Objects 採用（1位）

「1ルーム = 1 Durable Object」。ルームごとに単一スレッドの権威サーバーが立ち、全参加者が WebSocket で直結する。フェーズ遷移・投票・付箋の状態機械がこの1点で直列化されるため、レース条件が構造的に発生しない。ステルス投票は「開票前は本人にしか送らない」という選択的送信で表現でき、他人の票はネットワーク上を流れすらしない。

コスト
◎ WebSocket Hibernation でアイドル課金ゼロ。受信メッセージは 20:1 換算で課金、送信（fan-out）は無料。バースト利用 + 高頻度配信 + 長いアイドルというこのアプリの負荷特性に、課金モデルが唯一完全一致する。定常運用でも $0〜5/月（§04）。非商用制限もない。
ユーザビリティ
◎ 再接続 = DO が持つ現在状態のスナップショット送信、という最も素直な復帰パス。無料枠でもプロジェクトの自動停止がなく、「久しぶりに開いたら止まっていた」が起きない。
R3 適合
◎ push 型。ただし保証は手続き的コード＋テスト（@cloudflare/vitest-pool-workers）に依り、RLS のような宣言的な仕様書は残らない。
継続性
○ Cloudflare 本体は堅牢。DO 自体は独自機構でロックインは強め。ただし「ルーム = アクター」というモデル自体は移植可能な設計知識。
認証
Clerk（1万 MAU まで無料・マネージド）か Better Auth（OSS・D1 上に自前ホスト）。リスクを取らないなら Clerk。
C Convex + Clerk 次点（2位）

リアクティブ DB。クライアントは query / mutation というサーバー関数経由でしかデータに触れられないため、可視性制御のバイパス経路がそもそも存在しない。購読中のクエリは書き込みに応じて自動再実行され、楽観更新も組み込み。R1〜R3 だけならこのアプリに最も「はまる」開発モデル。

コスト
△ 弱点は R4。ドラッグ座標を mutation で流すと、書き込み + 参加者分のリアクティブ再実行が全て関数呼び出しとして課金され、1セッションで数十万コールに達しうる。Free（100万コール/月）は数セッションで尽き、実質 Pro（$25/開発者/月、2.5億コール込み）前提になる。
ユーザビリティ
◎ 一貫性・自動リアクティビティは最強クラス。再接続もクエリ購読の再確立だけで済む。
R3 適合
◎ 構造型。集計だけ返す開票クエリも自然に書ける。
継続性
△ スタートアップ依存。バックエンドは OSS 化済みでセルフホストの逃げ道はある。
A Next.js + Supabase + Vercel（現行構成） 3位 — ただし条件付きで1位

正データを Postgres の行として持ち、RLS で可視性を宣言し、broadcast_changes トリガーで差分配信する。PoC で中核リスク（ドラッグ同期・RLS の攻撃耐性）は実証済み。認可を「宣言 + pgTAP の回帰テスト」として資産化できる点は全候補で唯一であり、これは実装負荷ではなく保証の質の話なので、本評価でも正当に加点される。

コスト
△ 課金の単位が「メッセージ数」で、送受信の合計（fan-out 込み）が数えられる。ドラッグ配信が多いこのアプリでは Free の 200万msg/月 が月 7〜20 セッション程度で尽きる試算（§04）。加えて Vercel Hobby は非商用限定。
ユーザビリティ
△ 最大の問題は Free プロジェクトが1週間の放置で自動停止すること。「たまの週末にだけ使う学生アプリ」という利用パターンと最悪の相性で、回避には Pro（$25/月）が要る。同期品質そのものは PoC 実測で合格。
R3 適合
◎ pull 型。RLS + 列レベル GRANT + pgTAP。実攻撃 SQL で迂回不能を実証済み。
継続性
◎ 実体は Postgres。OSS でセルフホスト可能、コミュニティ最大。ロックインは全候補で最弱。
D Firebase（Realtime Database + Auth + Security Rules） 見送り

低レイテンシ同期・presence（onDisconnect）・オフライン耐性が組み込みで、R2 / R4 は得意。しかし R3 を Security Rules で書くと、開票時の集計に Cloud Functions が必要になり、rooms / notes / votes / groups というリレーショナルなデータを NoSQL ツリーに歪めて載せることになる。Rules は書き味が手続き的で、RLS のような SQL ベースの回帰テスト文化も薄い。コストは小規模なら数ドルで悪くないが、構造適合で B・C に勝る点がない。

E Liveblocks（+ 併設バックエンド） 見送り — ただし料金の誤解は正す

2026年時点の料金は MAU 課金ではなく月間アクティブ 500 ルームまで無料・MAU 無制限で、実はこのアプリの規模なら無料で回る。presence・カーソル・コメントの完成度は全候補で最高。それでも見送る理由は §02 そのもの——Storage はルーム内全員に複製される共有ドキュメントであり、ステルス投票と個人ワーク付箋は Liveblocks に置けない。核心機能のために結局サーバー + DB + 認可層を併設することになり、「最高の同期基盤が、いちばん大事な機能には使えない」という配置になる。自由なホワイトボード（Miro 型）を作るなら第一候補だが、PRD はそれを明確に否定している。

F 同期エンジン新世代（Zero / InstantDB / ElectricSQL） 見送り（時期尚早）

「権限付きの部分同期」という思想は R3 と方向が合っており、数年後の再評価では本命になりうる。現時点で見送る理由は成熟度が本番投入ラインに達しているか判断しづらいことと、ステルス投票の「開票までは本人のみ・開票後は集計のみ公開」という時間軸で変わる可視性が、どのみちサーバー関数を要求するため、優位が薄まること。

04コスト試算 — PoC の実測値から逆算する

絵空事にならないよう、PoC で実測した配信レート（80ms スロットル ≒ 12.5 msg/秒/ドラッグ中ユーザー）から1セッションの通信量を見積もる。前提: 6人 × 2時間、うち各人が計10分ドラッグ、付箋 CRUD・presence 込み。

  • ドラッグ送信: 12.5 msg/s × 600s × 6人 ≒ 4.5万 msg
  • Supabase の課金対象は送信 + 受信の合計。5人への fan-out を含めると 1セッション ≒ 20〜30万 msg
  • Cloudflare DO の課金対象は受信のみ（20:1 換算）、送信は無料。同じセッションが ≒ 2,250 課金リクエスト

ここが要点

差を生んでいるのは性能ではなく課金モデルと負荷特性の相性だ。このアプリのトラフィックは「たまに激しく、普段は無音」で、しかも1メッセージが5人に複製される。fan-out と待機時間に課金しない Cloudflare だけが、この形にぴったり重なる。

05ユーザビリティ比較 — 速さより「壊れないこと」

参加者は同じ会場にいることが多く、東京リージョン前提ならどの候補もネットワーク往復は体感差が出ない（PoC の 69〜99ms 間隔はスロットル由来であり回線由来ではない）。実際に体験を分けるのは次の3点。

観点B. Cloudflare DOC. ConvexA. Supabase
開いたら動くか ◎ 自動停止なし ◎ 自動停止なし △ Free は1週間放置で停止。復帰は手動
切断からの復帰 ◎ 再接続時に権威サーバーが現在状態を一括送信。設計に組み込み済みの概念 ◎ クエリ購読の再確立で自動復元 ○ 再購読 + スナップショット再取得を自前実装（PoC で実装済みのパターン）
進行の整合性（R1） ◎ 単一スレッド直列化でレース不能 ◎ トランザクショナルな mutation ○ DB トランザクションで担保（正しく書けば同等）

「久しぶりに開いたらプロジェクトが止まっていて、デモの直前に管理画面から再開する」——自動停止は障害ではなく仕様だが、利用者から見れば故障と区別がつかない。利用頻度が低いアプリほど、この一点がユーザビリティを支配する。

07推奨構成の具体形

ブラウザ (React SPA / Vite + TanStack Router + Tailwind + shadcn/ui)
  │  WebSocket（1本 / ルーム）
  ▼
Cloudflare Worker（静的アセット配信 + 認証ゲート: Clerk）
  │  ルーム ID でルーティング
  ▼
Durable Object  =  ルームの権威サーバー（東京 colo に自動配置）
  ├─ 状態機械: phase / step / タイマー (ends_at) / presence
  ├─ SQLite (DO 内蔵): notes / votes / groups / members
  └─ 選択的送信: 個人ワーク中の付箋・開票前の票は本人にのみ配信
        │
        ▼
D1: アカウント・ルーム一覧などルーム横断のデータ
  • Next.js は使わない。認証必須のツールで SEO が不要、ページはほぼ1枚のボード画面。SSR の便益がなく、SPA + 静的配信のほうがランタイムが単純で無料枠にも収まる。フレームワークの都合ではなく要件から導かれる結論で、これも「実装負荷を無視するなら」の帰結のひとつ。
  • ボード UI は引き続き自前実装。tldraw を使わない判断（docs/site の why-not-tldraw）はこの構成でも有効。tldraw SDK は無償利用だとウォーターマーク表示が条件で、外すには商用ライセンスが要る——コスト軸でも自前実装が正しい。
  • React + Storybook + MSW + zod + Vitest はそのまま活きる。変わるのはデータ層と認可層だけで、コンポーネント駆動の開発文化は構成に依存しない。
  • pgTAP の代替は「認可の否定系テスト」を DO に対して書くこと。@cloudflare/vitest-pool-workers で「member が他人の票を要求しても届かない」を回帰テストとして固定する。宣言的な RLS より仕様の見通しは落ちるため、可視性ルールを1つの純関数（visibleTo(state, userId)）に集約し、配信コードはそれ以外を経由できない構造にするのが要点。そうしないと、フィルタ漏れが即「他人の票が見える」事故になる。

08この判断が反転する条件

本メモの結論は前提に強く依存する。次のどれかが真になったら答えが変わる。

  • 実装負荷を評価に戻す — 認証は導入済み、RLS は攻撃 SQL まで検証済み、pgTAP 28件 + Vitest 50件が green。この資産を捨てる合理性はなく、現実の意思決定としては現行構成の継続が依然として正解。本メモは「理想形はどこか」を知った上でその判断を続けるためのもの。
  • 認可保証の宣言性を最優先に置く — 「ポリシーが仕様書を兼ね、SQL でテストできる」ことに最大の重みを置くなら、順位は A が1位に入れ替わる。push 型の保証はコード品質に依存する。
  • チームの学習目標がスタックを決める — 学校スクラムという文脈では「SQL・RLS・マイグレーションを学ぶこと」自体が成果物でありうる。教育的価値は実装負荷ではない別軸の便益で、これを重く見るなら A に加点される。
  • 収益化する — Vercel Hobby の非商用制限で A のコストに $20/席 が加算される一方、B は影響なし。差はさらに開く。
  • Miro 的な自由キャンバスへ方針転換する — R3 の重みが下がり、E（Liveblocks）や tldraw sync（Cloudflare テンプレート）が一気に浮上する。
