2 ── 設計方針

なぜ「throw ではなく戻り値」に全部書き換えたか

Next.js の公式ドキュメントにある規約です。「起こりうるエラー(入力ミス・未回答など)は throw せず、値として return せよ」。理由は throw した先の行き先にあります。

throw する(前の作り)

  1. 「まだ回答がありません」を throw new Error(...)
  2. Next.js はこれを「予期しないクラッシュ」として扱う
  3. 本番では詳細が隠され、汎用エラー画面に飛ばされる
  4. 入力した内容も画面の文脈も失われ、その場で直せない

戻り値で返す(今の作り)

  1. 「まだ回答がありません」を return { ok: false, message }
  2. ToastForm が結果を受け取る
  3. その場に留まったまま赤トーストで理由が出る
  4. 入力はそのまま。直してもう一度押せばいい

redirect を try の外に出した理由: Next.js の redirect() は実は「特殊な例外を throw して遷移する」仕組みです。try の中に置くと自作の catch がその例外を「エラー」と誤解して握りつぶし、遷移しなくなる。だからイベント作成などでは、DB 処理だけを try に入れ、redirect() は try の外に置いています。

3 ── 見つけたバグ

useEffect 版で「削除のトーストだけ出なかった」理由

初版の ToastForm は「結果を state に入れて、useEffect がそれを見てトーストを出す」作りでした。設定の保存では動いたのに、参加者の削除では出ない。違いは成功するとフォーム自身が消えるかどうかです。

初版(useEffect 方式)× 参加者削除

  1. 「外す」を実行 → サーバーで削除成功
  2. 結果と一緒に「新しい画面(その行なし)」が届く
  3. 行ごと ToastForm がアンマウント(消滅)
  4. トーストを出すはずの useEffect は、実行される前に持ち主ごと消えた → 何も出ない

現行(action 内で直接 toast)

  1. 「外す」を実行 → サーバーで削除成功
  2. 結果を受け取ったその場で toast() を呼ぶ(useEffect を待たない)
  3. sonner の toast() はコンポーネントの外にある掲示板への書き込みなので、直後にフォームが消えても影響なし
  4. 行が消え、トーストも出る

同じ理由で危なかった仲間: 「完了にする」(押すとボタン自体が消える)、「グループに投稿する」(押すと投稿済み表示に置き換わる)。現行方式ならすべて安全です。
