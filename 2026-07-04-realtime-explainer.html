前提

HTTP は「呼ばれたら答える」通信

あなたが慣れている HTTP は、いつも クライアントが口火を切ります。ブラウザがリクエストを送り、サーバーが一度だけ返事をして、接続は閉じる。次にほしい情報があれば、また新しいリクエストを送る。この「1往復で完結・サーバーからは勝手に喋れない」がすべての出発点です。

付箋アプリの「書き込み」側は、まさにこのモデルのままです。Server Action の中で supabase.from("notes").insert(...) を呼ぶと、内部では PostgREST への 普通の HTTP POST（POST /rest/v1/notes）が飛んでいるだけ。あなたが今まで書いてきた API 呼び出しと何も変わりません。

困りごと

HTTP だけで「相手の変更」を知る方法がない

問題はここです。他人がボードに付箋を貼ったのを、あなたのブラウザはどうやって知ればいい？ HTTP ではサーバーから勝手に通知できないので、素朴にやるなら「1秒ごとに聞きに行く」——これがポーリングです。

ポーリングは動くには動きますが、大半のリクエストが無駄で、サーバーとDBに常に負荷をかけ続け、それでいて反映は遅い。ドラッグ中の付箋をこの方式で追いかけるのは現実的ではありません。だから別の仕組みが要ります。

新しい道具

WebSocket = つなぎっぱなしの一本の線

WebSocket は、一度つないだら開けっ放しにする双方向の線です。毎回つなぎ直す HTTP と違い、線が生きている間はサーバーの側からいつでもメッセージを流し込めます。あなたが聞きに行かなくても、向こうから勝手に届く——これがポーリングとの決定的な差です。

HTTPユーザーへの橋渡し

実は WebSocket の始まりは HTTP リクエストそのものです。ブラウザは Upgrade: websocket ヘッダー付きの GET を送り、サーバーが 101 Switching Protocols を返す。あなたの知っているステータスコードで握手し、そこから先だけ「開けっ放しの線」に切り替わります。ゼロから知らない世界ではありません。

本題

①だけが従来のHTTP。②③は張りっぱなしのWebSocketに乗って自動で流れる

① 書き込みは今まで通りの HTTP

ブラウザA が付箋を動かすと、Server Action が DB に書き込みます。ここは普通の HTTP。

// app/rooms/actions.ts — 中身は PostgREST への HTTP POST/PATCH await supabase.from("notes").update({ x, y }).eq("id", noteId);

② DBのトリガーが「変わったよ」を投げる

ここが橋渡し役です。notes が変更されるたびにPostgres のトリガーが動き、その部屋あてのチャンネル（room:<ルームID>）へ変更内容を流します。アプリ側のコードは一切呼ばれません。DB が自分で投げます。

-- supabase/migrations の notes トリガー関数（抜粋） perform realtime.broadcast_changes( 'room:' || new.room_id, -- どのチャンネルへ tg_op, ... -- INSERT / UPDATE / DELETE と中身 );

③ Realtime が WebSocket で全員に配る

各ブラウザは最初にその部屋のチャンネルを購読しておきます。トリガーが流したメッセージを、Realtime サーバーが開きっぱなしの WebSocket 経由で購読者全員に push する。だからリロードなしで即座にブラウザBの画面に反映されます。

// app/rooms/[id]/room-board.tsx（抜粋） const channel = supabase.channel(`room:${roomId}`, { config: { private: true } }); channel .on("broadcast", { event: "INSERT" }, applyInsert) .on("broadcast", { event: "UPDATE" }, applyUpdate) .on("broadcast", { event: "DELETE" }, applyDelete) .subscribe(); // ← ここで WebSocket が張られる
要点

あなたが増やしたコードは「DBに書く」と「チャンネルを購読して届いたら画面に反映する」の2つだけ。WebSocketの張り方・再送・配信は Supabase 側が持っています。生の WebSocket API を自分で触る必要はありません。

セキュリティ

WebSocket にも RLS が効く

「開けっ放しの線」と聞くと、他人の部屋のメッセージまで受け取れてしまうのでは？と不安になります。ここで Supabase は、HTTP の DB アクセスと同じ RLS（行レベルセキュリティ）を WebSocket の配信にも使います。配信メッセージ自体が realtime.messages という保護されたテーブルを通るので、ポリシーで「そのルームのメンバーだけ受信可」を宣言できます。

-- room のメンバーだけが room:<id> チャンネルを受信できる create policy "room_members_can_receive_broadcast" on realtime.messages for select to authenticated using ( /* トピックの room に自分が所属しているか */ );

そのために、購読する前にログイン中のトークンを WebSocket に載せておく必要があります。これを忘れると、RLS が「未ログイン」と判断して購読が弾かれます。

await supabase.realtime.setAuth(); // ← 誰として繋ぐかを線に伝える。購読の直前に呼ぶ
今回このPoCで実際に直した穴

「メンバーなら誰でも更新OK」とだけ書いていたら、他人の付箋を自分のものに書き換えて盗む・消す攻撃が実際に成立しました。列単位で権限を絞り、攻撃を再現するテストで塞いだことを確認済みです。HTTP でも WebSocket でも、境界はDB側で守る——クライアントの表示制御だけに頼らないのが鉄則です。

応用

ドラッグ中だけは、DBを通さない

付箋をドラッグしている間の座標は、1回の操作で何十回も変わります。これを毎回 DB に書くと、書き込みも配信も溢れてしまう。そこでドラッグ中の座標だけは DB を経由せず、WebSocket でブラウザ間に直接流します。しかも約80ミリ秒に1回に間引き（スロットル）します。

Postgres は登場しない。指を離した瞬間だけ①のHTTPで最終位置を保存する
// 動かしている最中: DBではなくチャンネルへ直接送る（間引き付き） channel.send({ type: "broadcast", event: "note-drag", payload: { id, x, y } }); // 指を離したとき: ここで初めて HTTP で確定保存 await updateNotePosition(id, x, y);

つまり 「途中経過は軽い WebSocket、確定はHTTPでDBへ」 と役割を分けています。この設計のおかげで、実測でも受信間隔は 69〜99ms に収まり、遅延が積み上がらず滑らかに追従しました。

まとめ

あなたのコードのどこが HTTP で、どこが WebSocket か

最後に全体を1枚に。知っている HTTP はそのまま残っていて、WebSocket はその上に足された配信レイヤーだと分かれば、頭の中が整理できます。

ひとことで

書く・取るは HTTP のまま。「相手の変更が向こうから飛んでくる」ところだけが WebSocket。 その線を張るのは Supabase Realtime で、あなたは「購読する」「届いたら画面を更新する」を書くだけ。しかも RLS で守れる——だから安心して使えます。
