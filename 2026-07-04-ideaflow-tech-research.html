01

調査サマリー（推奨スタック）

結論: CRDT やホワイトボード SDK は不要。IdeaFlow に必要なのは「サーバー権威のリアクティブ同期」であり、第 1 推奨は React + Vite + Convex、第 2 推奨は Supabase 構成。MVP は無料枠内で運用できる。

なぜこの結論か

IdeaFlow には Google Docs 型の「同一テキストの同時編集」が存在しない。共同編集の対象は「付箋の追加・移動・グルーピング」という独立したオブジェクトの操作で、競合はほぼ起きない。一方で「ステルス投票」「フェーズ進行」「個人ワーク付箋の秘匿」というサーバー側でしか実現できない情報の出し分けが 3 つもある（詳細は §03）。この 2 点から、CRDT（Yjs）やコラボ SDK（Liveblocks）ではなく、サーバー権威モデルの BaaS が設計に一致する。

そうしないとどうなるか: CRDT/共有ドキュメント型を選ぶと、全データが全クライアントに複製されるため、投票の秘匿や個人付箋の非公開を原理的に実現できず、結局サーバー側 API を別途構築することになり二重投資になる。

02

PRD 要件の技術分解

PRD のコア機能一覧（§7）を技術要件に分解した。右列の分類が、後続のアーキテクチャ選定の判断軸になる。

PRD のコア機能技術要件分類
ルーム作成・招待コード参加 匿名認証、短い招待コードの発行・照合、メンバー一覧のリアルタイム反映（Presence） 共有状態の同期
デザインスプリント進行フロー（5 フェーズ × Stepper） ルーム単位の状態機械（phase / step）。ホストの操作で全員の画面が同時遷移する サーバー権威
個人ワーク付箋 + タイマー 本人にだけ見える付箋（visibility 制御）、全員で同期するカウントダウン 情報の出し分け
共有・グルーピング 付箋のドラッグ移動・グループ割当のリアルタイム同期。ドラッグ中の座標は高頻度・低重要度 共有状態の同期
ステルス投票（主観×1・客観×3） 投票中は自分の投票だけ見え、他人の投票はデータとしても届かないこと。決定ステップで集計を公開 情報の出し分け
2 軸マッピング（価値 × 実現難易度） ホストのみドラッグ可能という権限制御をサーバー側で強制 サーバー権威
発想支援ツール / ファシリテーションガイド 静的コンテンツ（固定文言）。フェーズ・ステップに応じて出し分けるだけで、バックエンド不要 フロントのみ

分類すると、「サーバー権威」と「情報の出し分け」が全体の半分以上を占める。逆に、自由描画・図形・コネクタといったホワイトボード固有の要件はゼロであり、PRD の Non-Goal（多機能ホワイトボードは目指さない）とも一致する。AI/LLM 連携も Non-Goal のため、推論基盤の調査は対象外とした。

03

最重要の設計判断: ステルス投票がアーキテクチャを決める

今回の調査で最も重要な発見は、ステルス投票の要件がリアルタイム基盤の選択肢を強く絞り込むことである。

リアルタイム共同編集の実現方式は大きく 2 系統に分かれる。

方式代表仕組みステルス投票との相性
共有ドキュメント型（CRDT） Yjs、Liveblocks Storage、tldraw sync 1 つのドキュメントを全クライアントに複製し、変更を自動マージする ✕ ドキュメントに入れたデータは全員に届く。UI で隠しても DevTools で他人の投票が見える
サーバー権威型 Convex、Supabase（RLS）、Firebase（Security Rules） データはサーバー側にあり、クライアントには閲覧を許可した分だけを配信する ◎ 「投票中は自分の票のみ」「決定ステップ以降は集計を全員に」をサーバー側で強制できる
そうしないとどうなるか

「投票データも Yjs ドキュメントに入れて、UI 側で他人の票を非表示にする」実装は一見動くが、ブラウザの開発者ツールを開けば全員の投票がリアルタイムで丸見えになる。ステルス投票は PRD の差別化要素（発言力に左右されない意思決定）そのものなので、「見えないふり」ではなくデータレベルの秘匿が必要。同じ理屈が「個人ワーク中の付箋」にも当てはまる。

同様に、フェーズ進行・ホスト専用操作（2 軸マッピングのドラッグ、フェーズ確定）も、クライアント側のチェックだけでは改ざん可能なため、サーバー側の書き込み検証（「この操作はホストか？」「今このフェーズか？」）が必要になる。つまり IdeaFlow は 「コラボアプリに見えて、実態はサーバー権威のターン制アプリに近い」。これが基盤選定の指針となる。

A. Convex — 第 1 推奨

TypeScript でクエリ・ミューテーションを書くと、データ変更時に購読中の全クライアントが自動更新されるリアクティブ BaaS。「ホストが Step を進めたら全員の画面が変わる」「誰かが付箋を出したらボードに現れる」という IdeaFlow の中核体験が、購読 (useQuery) を書くだけで成立する。投票の秘匿も「query 関数が返すデータを絞る」だけで実現でき、§03 の要件と完全に一致する。ミューテーションがトランザクショナルなので、「全員がほぼ同時に最後の 1 票を入れる」ような競合も安全。懸念だったロックインは、バックエンドが OSS 公開（FSL ライセンス）されセルフホスト可能になったことで緩和されている。SQL が使えない点は、MVP のデータモデル（後述、6 テーブル程度）では制約にならない。

B. Supabase — 第 2 推奨（こちらを選ぶ条件つき）

Postgres + Row Level Security（RLS）で「投票中は自分の行しか読めない」を宣言的に書ける点は Convex と並んで優秀。匿名認証（signInAnonymously）が公式機能として最も成熟しているのも利点。弱点は、リアルタイム反映を Postgres Changes / Broadcast の購読とクライアント側の再取得を自分で設計する必要があり、Convex 比で配線コードが増えること。SQL・RDB 設計を学びたい、将来セルフホストしたい、日本語の情報量を重視するならこちらを選ぶ価値がある。

C. Liveblocks — 不採用

Presence・共同編集の DX は最高水準だが、Storage（共有状態）が全クライアント複製型のため、ステルス投票・個人付箋・ホスト権限の 3 要件すべてで別途バックエンドが必要になる。「コラボ SDK + 自前 API」の二重構成になるくらいなら、最初からサーバー権威型 1 本にまとめる方が構成が単純。テキスト共同編集（コメント・ドキュメント）を将来足すことになったら、その時に部分導入すればよい。

D. Yjs + PartyServer（Cloudflare Durable Objects）— 不採用

PartyKit は Cloudflare に買収され、後継の PartyServer + Durable Objects として健在。ルーム = Durable Object という対応付けは IdeaFlow のモデルと相性が良く、技術的には最も自由度が高い。ただし認証・永続化・秘匿ロジック・再接続処理をすべて自作することになり、MVP の価値検証（型に沿えば迷わず決められるか）に到達するまでの距離が最も遠い。インフラを学ぶ副教材としては魅力的だが、プロダクト検証が目的なら過剰投資。

なぜ Convex を第 1 推奨にしたか

比較表の上 3 行（秘匿・サーバー権威・リアルタイム反映コスト）が IdeaFlow の成立条件であり、3 つすべてが ◎ なのは Convex だけだから。特に「リアルタイム反映の実装コスト」は開発期間に直結する。Supabase 案は成立するが、購読・リフェッチの配線という「プロダクト価値と無関係な作業」が MVP 期間を圧迫する。

05

付箋ボード（キャンバス）の実装方式

「共有・分類」ステップのボードと 2 軸マッピングをどう作るかを比較した。前提として、必要なのは付箋のドラッグ・グルーピング・ドット表示のみで、自由描画・図形・無限キャンバスは Non-Goal である。

方式概要評価
DOM + dnd-kit（推奨） 付箋を position: absolute の div として描画し、dnd-kit でドラッグ制御。2026 年時点で React の DnD ライブラリのデファクトで、活発にメンテされている ◎ 付箋は数十〜百枚規模で DOM で十分。テキスト入力・IME・アクセシビリティが無料で手に入る。2 軸マッピングも同じ実装を座標系だけ変えて再利用できる
react-konva（Canvas 描画） HTML5 Canvas 上に付箋を描画 △ 数千オブジェクト規模なら優位だが、テキスト編集・IME 入力を自前実装する必要があり、付箋アプリでは弱点が直撃する
tldraw SDK + tldraw sync 完成度の高いホワイトボード SDK ✕ 本番利用は商用ライセンス（年額 $6,000）または申請制の Hobby ライセンスが必要。かつ全機能入りホワイトボードは Non-Goal と衝突し、sync は共有ドキュメント型なので §03 の問題も抱える
なぜ Canvas ではなく DOM か

付箋の本質は「テキストの入力と表示」であり、これは DOM が最も得意とする領域だから。Canvas 系を選ぶと、日本語 IME での入力、テキストの折り返し、スクリーンリーダー対応をすべて自前で作り直すことになる。IdeaFlow の付箋枚数（5 人 × 10 枚前後 × フェーズ）では Canvas の描画性能優位は一切効いてこない。

06

個別設計ポイント

6-1. フェーズ / ステップ進行（Stepper）

ルームの現在地は rooms.phase / rooms.step の 2 フィールドをサーバー側の単一の真実とする。ホストが「次へ」を押すとミューテーションが権限（ホストか）と遷移の妥当性（決定前に飛ばしていないか）を検証して更新し、全クライアントは購読経由で同時に画面遷移する。クライアント側に XState などの状態機械ライブラリを入れる必要はなく、サーバーの値を描画するだけの「読み取り専用ステート」に保つのが肝。

6-2. タイマー同期

「残り 3:00」をネットワークでカウントダウン配信するのではなく、終了時刻 timerEndsAt（サーバー時刻）だけを保存し、各クライアントがローカルで残り時間を計算・描画する。クライアントとサーバーの時計ズレは、接続時にサーバー時刻とのオフセットを一度取得して補正する。そうしないと、毎秒のブロードキャストで帯域を浪費するうえ、遅延で端末ごとに表示がズレる。

6-3. ドラッグ座標の同期（高頻度更新の逃がし方)

付箋のドラッグ中の座標を毎フレーム DB に書くと、書き込み回数が爆発する。「ドラッグ中は ephemeral（一時）チャネルに間引いて送信（50〜100ms スロットル）、ドロップ時に確定座標を 1 回だけ DB に書く」という 2 層構成にする。Convex なら presence 系コンポーネントまたはスロットルしたミューテーション、Supabase なら Broadcast チャネルが ephemeral 層に該当する。他人のドラッグがカクついても実害はないが、確定位置がズレると議論に影響するため、この非対称性を利用した設計。

6-4. 個人ワーク付箋の可視性切替

付箋に visibility: "private" | "shared" を持たせ、private の間はサーバー側で作者本人にしか返さない。共有ステップへの遷移時に一括で shared に切り替える。「個人ワークで安心して書ける」という PRD の価値は、このデータレベルの秘匿で担保される（§03 と同じ原理）。

6-5. 招待コードと匿名参加

参加フローは「6 文字コード入力 → 匿名認証 → 表示名入力 → 入室」。コードは紛らわしい文字（0/O、1/I/l）を除いた 31 文字セットで 6 桁 ≒ 8.9 億通りとし、総当たりを流量制限で防ぐ。アカウント登録を要求しないことが、ハッカソン現場での「全員が 30 秒で入室できる」体験に直結する。ルームには TTL（例: 72 時間）を設け、無料枠のデータ量を守る。

6-6. データモデル（叩き台）

rooms      { id, code, hostId, phase, step, timerEndsAt, createdAt }
members    { roomId, userId, displayName, role: "host" | "member" }
notes      { roomId, authorId, phase, text, x, y, groupId?,
             visibility: "private" | "shared" }
groups     { roomId, phase, label }
votes      { roomId, phase, targetId, voterId,
             kind: "subjective" | "objective" }   // 🔴×1 / 🔵×3 の上限はサーバー側で検証
decisions  { roomId, phase, targetId }            // 各フェーズの確定成果物

6 テーブルで MVP 全機能が表現できる。HMW・アイデア・課題はいずれも notes の phase 違いとして扱えるため、フェーズごとに専用テーブルを作らない。2 軸マッピングは notes の x/y を「価値 × 実現難易度」座標として解釈し直すだけで、ボードと同じ仕組みに乗る。

推奨アーキテクチャ

クライアント — React 19 + TypeScript + Vite（SPA）
付箋ボードDOM + dnd-kit / 2 軸マップと共通実装
Stepper + ガイド文言サーバーの phase/step を描画するだけ
投票 UI主観×1 客観×3
タイマー表示endsAt からローカル計算
発想支援サイドバー静的コンテンツ（SCAMPER 等）
ローカル UI 状態Zustand（ドラッグ中・入力中のみ）
useQuery: リアクティブ購読（WebSocket） mutation: 付箋 CRUD・投票・フェーズ遷移
バックエンド — Convex（サーバー権威）
queries閲覧権限に応じた出し分け（秘匿はここで実現）
mutations権限・票数上限・遷移妥当性の検証（トランザクショナル）
presence参加者一覧・ドラッグ中座標（ephemeral）
schedulerタイマー終了処理・ルーム TTL 掃除
DBrooms / members / notes / groups / votes / decisions

配信は Vercel または Cloudflare Pages の無料枠（SPA の静的配信のみ）。SSR が不要なのは、アプリ全体が招待コードの先にありSEO 対象ページが実質ランディング 1 枚だからで、Next.js を採用する動機が薄い。フレームワークを薄く保つことで、コントラクト層（Convex のスキーマと関数シグネチャ）を厳密に定義し、UI 実装側を再生成可能に保つ方針とも整合する。

Supabase 構成に切り替える場合の差分

クライアントはほぼ同じ。バックエンド層が「RLS ポリシー（秘匿）+ DB 関数（検証つき書き込み）+ Realtime 購読（Postgres Changes / Broadcast / Presence）」の組み合わせに置き換わる。追加で必要になるのは、変更通知を受けてからのクエリ再取得の設計（TanStack Query 併用を推奨）。

08

コスト試算（学生チーム規模）

想定負荷: 1 ルーム 5 人前後、同時アクティブ 1〜20 ルーム、1 セッション 1〜2 時間。この規模では全構成が無料枠に収まる。

Supabase 同時リアルタイム接続 200 / 月 200 万メッセージ / DB 500MB Pro: $25/月（接続 500 まで込み、超過 $10/1,000 接続） 同時 200 接続 = 5 人ルーム × 40。検証には十分

Vercel / Cloudflare Pages 個人・Hobby 無料枠 商用化時 SPA 静的配信のみなので負荷ほぼゼロ
tldraw SDK（参考） 開発・Hobby（申請制）は無料 商用ライセンス 年額 $6,000 不採用の根拠の一つ

結論として、MVP の価値検証フェーズで金銭コストは発生しない。コストよりも、無料枠の「同時接続数」がハッカソン当日のような瞬間ピークで頭打ちにならないかだけ注視すればよい（§09 リスク参照）。

09

リスクと対策

リスク影響対策
Convex のベンダーロックイン 将来の移行コスト バックエンドは OSS（FSL）でセルフホスト可能。さらにデータモデル（§6-6）を Postgres でも表現可能な形に保ち、ドメインロジックを関数として分離しておく
ハッカソン当日の瞬間ピークで無料枠の同時接続を超過 新規入室不可 イベント利用が見えた時点で 1 ヶ月だけ Pro（$25）に上げる。両候補とも段階課金なので突発コストは小さい
招待コードの総当たり・荒らし入室 ルーム荒らし コード空間 8.9 億通り + 参加試行の流量制限 + 匿名認証への CAPTCHA（Supabase は公式推奨）。ホストによる強制退出機能を早期に用意
ドラッグ同期の書き込み過多 枠消費・カクつき §6-3 の ephemeral + 確定書き込みの 2 層構成。最悪、ドラッグ中の中継を諦めてドロップ結果だけ同期しても UX は成立する
匿名ユーザーのセッション消失（リロード・回線切断） 付箋・投票の紐付けが切れる 匿名認証はトークンが localStorage に残るため通常のリロードは復帰可能。加えて「同じ表示名での再入室時に旧メンバーへ再紐付け」する救済フローを用意
全員同時操作での競合（付箋の同時移動・最後の 1 票） データ不整合 付箋は基本 last-write-wins で実害なし（独立オブジェクトのため）。票数上限・フェーズ遷移だけはトランザクショナルな mutation / DB 関数で厳密に検証
10

実装ロードマップ（MVP）

PRD のフロー順ではなく、技術リスクが高い順（同期 → 秘匿 → 装飾）に検証する並びにした。仮に基盤選定に誤りがあっても Step 1〜2 で顕在化する。

  1. STEP 1
    ルーム基盤 — 作成 / 招待コード / 匿名参加 / メンバー一覧（Presence） リアルタイム購読と匿名認証という 2 大リスクをここで検証。5 人で同時入室して名前が即座に並べば基盤選定は正しい
  2. STEP 2
    フェーズ進行 — phase/step のサーバー権威 + Stepper UI + ガイド文言表示 ホスト操作で全員の画面が同時遷移することを確認。ガイド文言は静的データなのでこの時点で全フェーズ分入れられる
  3. STEP 3
    付箋 — 個人ワーク（private）+ タイマー + 共有切替 + ボードのドラッグ / グルーピング visibility 切替と ephemeral 座標同期という、このプロダクト固有の設計が正しく動くかの検証
  4. STEP 4
    ステルス投票 — ×1 / ×3 の上限検証 + 集計公開 + 決定ステップ 「投票中は他人の票がデータとして届かない」ことを DevTools で確認するテストを必ず入れる
  5. STEP 5
    仕上げ — 2 軸マッピング（ホスト専用ドラッグ）+ 発想支援サイドバー + 全 5 フェーズ通し 2 軸マップは Step 3 のボード実装の座標系違い。最後に 5 人で①→⑤の通しリハーサルを行い、PRD の Goal 1（迷わず完走）を検証
