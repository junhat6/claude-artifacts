全体像：1本のスコープチェーン

発展要件 5（UI/UX 改善）。スキーマ変更なし — 既存テーブルのまま、クエリとビューだけで完結

既存の検索（search）・タグ絞り込み（with_tag）に、今回の sorted（並べ替え）と page（ページ分割）をスコープチェーンの末尾に足しただけで、4つの機能が任意の組み合わせで同時に効く。Active Record の Relation が遅延評価で合成される性質をそのまま使っている。

app/controllers/books_controller.rb（index — 変更後の全文）ruby
def index
  # 存在しない tag_id は絞り込みなしとして扱う（フィルタ表示も出さない）
  @current_tag = Tag.find_by(id: params[:tag_id])
  @books = Book.search(params[:q]).with_tag(@current_tag&.id)
               .includes(:authors, :tags, copies: :rentals)
               .sorted(params[:sort], params[:direction])
               .page(params[:page])
end

なぜこの形か

検索・絞り込み・並べ替え・ページ分割をそれぞれ独立したスコープにしておくと、機能の掛け算（検索×ソート×2ページ目、など）がコードの掛け算にならない。もしコントローラに if params[:sort] の分岐で order を直書きしていくと、条件の組み合わせごとに分岐が爆発し、テストも組み合わせの数だけ必要になる。

Relation は遅延評価（実際に要素へアクセスするまで SQL を発行しない）なので、4段チェーンでもクエリは最終的に1本の SELECT に合成される。「スコープを重ねる＝クエリが増える」ではない、はレビューで補足できると良い。

なぜ kaminari か（pagy・自作との比較）

kaminari：page がスコープとして生えるので既存の search / with_tag / sorted と同じ文法で繋がる。pagy は最速・最軽量だが @pagy, @records = pagy(scope) という独自のコントローラ API でスコープチェーンから外れる。自作（limit/offset）は仕組みの学習にはなるが、総ページ数計算・境界処理・リンク生成を全部自前で正しく書く必要があり、車輪の再発明になる。学習リソースの多さも含めて kaminari を選択（実装前にユーザーと合意）。

per_page はデフォルトの 25 から 10 に設定。この課題のシードデータ（150冊）で16ページになり、ページャの見た目・境界の挙動を確認しやすい粒度にしている。

検証で見つけて直したバグ：範囲外ページ
curl での画面検証中に ?page=999 を直接叩くと、153冊あるのに「本はまだ登録されていません」と誤表示されることを発見（@books.any? が false になり未登録の分岐に落ちる）。テストを先に書いて Red を確認してから、@books.total_count.positive? の分岐を追加し「このページには本がありません＋1ページ目へ戻る」の案内に修正した。空配列には「0件」と「範囲外」の2つの意味がある、という学びが残る不具合。

02

ソート機能 — 許可リストと順序の決定性

発展要件 5 前半。「スコープはモデルに置く」既存パターン（search / with_tag）に整合

タイトル・出版年をテーブルヘッダのクリックで昇順⇄降順にトグルできる。短いスコープだが、order インジェクション対策（許可リスト）とページネーションと安全に共存するための第2ソートキーという2つの設計判断が詰まっている。

app/models/book.rb（scope :sorted）ruby
# params 由来の値を order に直接渡すと SQL インジェクションになるため、
# 許可リスト外のカラムは完全デフォルト（登録日降順）に落とす。
# id の第2キーは、同値キーでもページ跨ぎの重複・欠落が起きないよう全順序を確定させるため
SORTABLE_COLUMNS = %w[title published_year].freeze

scope :sorted, ->(column, direction) {
  if SORTABLE_COLUMNS.include?(column.to_s)
    dir = %w[asc desc].include?(direction.to_s) ? direction.to_sym : :asc
    order(column.to_s => dir, id: :desc)
  else
    order(created_at: :desc, id: :desc)
  end
}
app/helpers/books_helper.rb（全文）ruby
module BooksHelper
  # 一覧の絞り込み・並び替え状態を URL 間で引き継ぐための共通パラメータ。
  # page を含めないのは意図的：絞り込みや並び順が変わると同じページ番号は
  # 別の中身を指すため、常に1ページ目からやり直す
  def preserved_list_params(overrides = {})
    {
      q: params[:q].presence,
      tag_id: params[:tag_id].presence,
      sort: params[:sort].presence,
      direction: params[:direction].presence
    }.merge(overrides).compact
  end

  def sort_link(column, label)
    active = params[:sort] == column
    # 不正な direction はモデル側で昇順として扱うため、表示・トグルも昇順起点に揃える
    current_direction = params[:direction] == "desc" ? "desc" : "asc"
    next_direction = active && current_direction == "asc" ? "desc" : "asc"
    arrow = active ? (current_direction == "asc" ? " ▲" : " ▼") : ""

    link_to "#{label}#{arrow}",
            books_path(preserved_list_params(sort: column, direction: next_direction)),
            class: "link link-hover"
  end
end
なぜこの設計か（2つの罠と対策）

① 許可リスト（ホワイトリスト）方式 — order(params[:sort]) と直接渡すと、?sort=isbn);DROP TABLE books;-- のような値がそのまま SQL に埋め込まれ得る（order はプレースホルダで守れない位置）。カラム名を SORTABLE_COLUMNS と照合し、一致しなければユーザー入力を一切 SQL に触れさせないのが唯一確実な対策。検証には Arel や order のハッシュ形式（order(title: :asc)）を使い、文字列連結を避けている。

② id を第2ソートキーにする — 出版年ソートでは同年の本が大量にある。ORDER BY のキーが同値の行同士は SQL 標準では順序が保証されないため、1ページ目と2ページ目で並びが揺れ、同じ本が両ページに出る／どのページにも出ないという不具合がページ境界で起こる。一意な id を最後に足して全順序を確定させるのが、LIMIT/OFFSET 型ページネーションの定石。

③ ソート変更時に page を捨てる — 並び順が変わった後の「3ページ目」は変更前とは別の中身を指す。意味を失ったページ番号を引き継ぐより、1ページ目に戻すのが正しい UX（preserved_list_params が page を含めない理由）。

テストの見どころ
test/models/book_test.rb では Book.sorted("isbn); DROP TABLE books;--", "asc") と攻撃文字列そのものを渡してデフォルト順にフォールバックすることを検証。さらに created_at を同時刻に固定した2冊を作り、同値キーでも順序が安定する（id 降順になる）ことを別テストで固定している。正常系は「作った2冊の相対位置」で検証し、実装の order をなぞるだけの偽陽性テストを避けた。

なぜカード切替か・なぜパーシャル化か

カード切替 vs 横スクロール：8列のテーブルをスマホで横スクロールさせると、1行の情報を読むのに毎回スワイプが要り、操作ボタンも画面外に隠れる。カードなら1冊分の情報＋操作が1画面幅に収まる。代償としてマークアップが2系統になるが、これは次のパーシャル化で緩和した（実装前にユーザーと合意した選択）。

パーシャル化（_stock_badge / _actions）：出し分けの最大のリスクは片方だけ直して片方を忘れる二重管理。在庫バッジの色分岐や管理者だけに出す削除ボタンのような「ロジックを含む断片」を共用パーシャルに寄せておけば、修正は常に1箇所で済む。そうしないと、例えば削除ボタンの権限条件を変えたときテーブル側だけ直り、カード側に古い権限のボタンが残るという事故が起こる。

コントローラは1つ・@books も1つで、変わるのは見せ方だけ。状態とロジックはそのまま、プレゼンテーション層だけが分岐するという関心の分離を守っている。

想定問答 — 強みと、正直な弱点

レビュー会で聞かれそうな質問と回答の準備。強みは胸を張って、改善余地は正直に + 「やるならこうする」をセットで

まず押さえる3つの強み

  • order インジェクション対策params を order に直接渡さず許可リスト照合。攻撃文字列を渡すテストまである（→ 02）
  • 順序の決定性id を第2ソートキーにして、同値キーでのページ跨ぎ重複・欠落を構造的に防止（→ 02）
  • スコープ合成で機能が掛け算検索×タグ×ソート×ページが分岐ゼロで同時に効く。既存の search / with_tag の設計に素直に乗った（→ 00）

想定問答（クリックで回答を開く）

Q1URL の sort パラメータに SQL を書き込まれたらどうなりますか？

強み何も起こりません。SORTABLE_COLUMNS（title / published_year）との照合に失敗した値は一切 SQL に触れずにデフォルトの登録日降順へフォールバックします。order 句はプレースホルダで守れない位置なので、「検証して使う」ではなく「一致しなければ使わない」許可リスト方式が唯一確実です。テストでは "isbn); DROP TABLE books;--" という攻撃文字列そのものを渡してフォールバックを検証しています。

Q22ページ目に1ページ目と同じ本が表示されることはありませんか？

強み起こらない設計です。ORDER BY のキーが同値（例：同じ出版年）の行は SQL 標準では順序が保証されず、クエリごとに並びが揺れて LIMIT/OFFSET のページ境界で重複・欠落が起こり得ます。そこで sorted は常に一意な id を第2キーに足して全順序を確定させています。created_at を同時刻に固定した2冊で順序が安定することを検証するテストもあります。

Q3検索しながらソートして3ページ目、のような組み合わせは正しく動きますか？

強み動きます。4機能がそれぞれ独立したスコープで、コントローラは search(...).with_tag(...).sorted(...).page(...) と機械的に繋ぐだけです。URL 間の状態引き継ぎは preserved_list_params ヘルパーに集約し、ページリンクは kaminari が現在のパラメータを自動継承します。なおソートや検索を変えたときは意図的に1ページ目へ戻します。並び順が変わった後の「3ページ目」は別の中身を指すため、ページ番号を引き継ぐ方がバグだからです。

Q4日本語タイトルは五十音順に並びますか？

改善余地正直に言うと、完全な五十音順にはなりません。SQLite の既定の照合順序（バイナリ照合）では文字コード順になるため、ひらがな・カタカナ・漢字が混ざると辞書順とはずれます（英字も大文字→小文字の順）。やるなら：①読み仮名カラム（title_reading）を追加してそれをソートキーにする（入力コスト増だが確実）、② ICU 照合を持つ PostgreSQL へ移行し COLLATE "ja-JP-x-icu" を使う、の2択です。書誌データとして正確なのは①で、図書館システムの実務でも読み仮名を持つのが定石、と答えます。

Q5ソートやページ移動のたびに全ページ再読み込みするのですか？

改善余地現状は通常のリンク遷移で、Turbo Drive が body を差し替えています（体感は十分速いですが、スクロール位置は先頭に戻ります）。やるなら：一覧を turbo_frame_tag "books_list" で包み、ソートリンクとページャに data: { turbo_frame: "books_list" } を付ければ、一覧のテーブル部分だけが部分更新されます。Rails 8 なら追加 gem なし・JS 数行以下で実現でき、次のスプリント候補です。

Q6テーブルとカードを両方描画するのは無駄ではないですか？

改善余地ご指摘のとおり、DOM には常に2系統が存在し、片方は CSS で隠れているだけです。1ページ10冊に絞ってあるので実害（描画コスト・転送量）は小さいものの、二重描画自体は事実です。やるなら：①マークアップは1系統にして CSS Grid で「md 未満ではセルをカード風に再配置する」スタイルだけで解決する（HTML は減るが CSS が複雑化）、② User-Agent でサーバー側から出し分ける（キャッシュと相性が悪い）。①と現状の折衷が現実的で、共通部分をパーシャルに寄せて二重管理のリスクだけ先に潰したのが今回の判断です。

Q7データが増えたとき、このページネーションの限界はどこですか？

改善余地2つあります。①total_count の COUNT クエリが毎リクエスト走る（件数バッジとページ数計算に必要）。②OFFSET 方式は深いページほど遅い（OFFSET 10000 は1万行読み捨てる）。蔵書数万冊規模なら、COUNT を省略できる pagy の countless モードや、「id より小さい次の10件」を辿るカーソル（キーセット）ページネーションに切り替えます。カーソル方式が成立するのは全順序が確定しているときだけなので、id 第2キーの設計（Q2）はその布石にもなっています。

Q8なぜ pagy ではなく kaminari を選んだのですか？

説明できる選択性能ベンチマークでは pagy が優位なことは把握していますが、この規模（数百冊）で差は体感できず、選定基準を「既存設計との整合」に置きました。kaminari の page は Relation を返すスコープなので、このアプリの核である search → with_tag → sorted のスコープチェーンに同じ文法で繋がります。pagy はコントローラで @pagy, @records = pagy(...) と受ける独自 API で、チェーンの外に出ます。学習教材・日本語情報の豊富さも kaminari が上、という判断です（実装前に選択肢と推奨を提示して合意済み）。
