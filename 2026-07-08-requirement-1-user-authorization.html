00

全体像 — 認証と認可の二段ゲート

発展要件 1（ユーザー権限）に対応。画面遷移図の「書籍登録（管理者のみ）」「書籍編集/削除（管理者のみ）」を実装で回収

書き込み系アクションへのリクエストは2つの before_action ゲートを順に通過する。ゲート1が認証（誰なのか）、ゲート2が認可（何をしてよいか）。users.admin カラム（boolean・NOT NULL・default false）は前スプリントで DB に用意済みだったため、本 PR はマイグレーションなしの純粋な認可層の追加になっている。

ゲート1require_authentication認証：誰なのか（既存）

Authentication concern が全アクションを既定で保護。未ログインは /session/new へリダイレクトされ、ここで止まる。通過すると Current.user が確定する。

ログイン済みユーザーだけが通過（Current.user が使える状態になる）
ゲート2require_admin認可：何をしてよいか（本PRで追加）

Authorization concern が提供。admin? が false（一般ユーザー）なら root へリダイレクトし、alert「この操作は管理者のみ行えます。」を表示する。

admin? == true だけが通過
到達アクション実行new / create / edit / update / destroy

書籍の登録・編集・削除に到達できるのは管理者のみ。閲覧系（index / show）と貸出・返却はこのゲートの対象外。

実装後の権限マトリクス

操作ゲスト（未ログイン）一般ユーザー管理者
一覧・詳細・検索○○○
貸出・返却× ログイン画面へ○○
書籍の登録・編集・削除× ログイン画面へ× root へ + alert○

要件の文言は「登録・削除」だが、「一般ユーザーは貸出・返却のみ可能」と画面遷移図の「書籍編集/削除（管理者のみ）」から、編集（edit / update）も管理者限定と解釈した（→ 想定問答 Q3）。

01

Authorization concern — 認可の置き場所

新規ファイル app/controllers/concerns/authorization.rb（全16行）

既存の Authentication concern と同じ流儀（concern + private メソッド + helper_method）で認可を実装した。Pundit / CanCanCan のような認可 gem は使っていない。以下が新規ファイルの全文。

app/controllers/concerns/authorization.rb（全文）ruby
module Authorization
  extend ActiveSupport::Concern

  included do
    helper_method :admin?   # ビューからも admin? を呼べるようにする
  end

  private
    def admin?
      authenticated? && Current.user.admin?
    end

    def require_admin
      redirect_to root_path, alert: "この操作は管理者のみ行えます。" unless admin?
    end
end
app/controllers/application_controller.rbruby
class ApplicationController < ActionController::Base
  include Authentication
  include Authorization   # 全コントローラ・全ビューで admin? / require_admin が使える

活用した Rails の機能

  • ActiveSupport::Concern
    include 時のフック（included do）を宣言的に書ける。認証と同じ構造で「関心ごとに1ファイル」を保つ。
  • helper_method
    コントローラの private メソッドをビューにも公開。同じ admin? が「ガード」と「ボタン出し分け」の両方の判定源になる。
  • ActiveSupport::CurrentAttributes
    Current.user.admin? で現在ユーザーの権限をどこからでも参照。リクエスト毎に自動リセットされる。
  • 属性の述語メソッド
    boolean カラム admin には admin? が自動生成される。モデルに1行も足さずに済んでいる。
なぜこの設計か

認証と認可を別 concern に分けた：「誰であるか（Authentication）」と「何をしてよいか（Authorization）」は変更理由が異なる関心事。1つの concern に混ぜると、将来ロールが増えたときに認証のコードまで触ることになり、変更の影響範囲が読めなくなる。

認可 gem を使わなかった：権限が admin boolean 1つで、リソース別のポリシークラス（Pundit の BookPolicy など）を張るほどの複雑さがない。既存の認証も gem なしの Rails 8 標準構成なので、認可だけ gem に頼るとコードベース内に2つの流儀が混在する。学習課題として「全行説明できる」ことも優先した。

admin? が authenticated? を先に評価する：未ログイン時は Current.user が nil なので、直接 Current.user.admin? と書くと公開ページ（一覧・詳細）のビューで NoMethodError になる。短絡評価で「未ログイン = 管理者ではない」に安全に倒している。

02

クローズ既定の適用 — except で閉じる

app/controllers/books_controller.rb への追加は1行

BooksController への適用は before_action 1行。ただし only: ではなく except: を選んだところに設計判断がある。公開アクションを列挙して「それ以外は全部ロック」という向き＝クローズ既定（closed by default）。

app/controllers/books_controller.rb（冒頭）ruby
class BooksController < ApplicationController
  allow_unauthenticated_access only: [ :index, :show ]      # 認証ゲートの opt-out（既存）
  before_action :require_admin, except: [ :index, :show ]   # 認可ゲート（本PRで追加）
  before_action :set_book, only: [ :show, :edit, :update, :destroy ]
なぜこの設計か（そうしないとどうなるか）

2行が鏡像になっている：「index / show は未ログインでも見られる」×「index / show 以外は管理者のみ」。公開範囲の定義が同じリスト [ :index, :show ] で2回現れるので、公開面がどこかはこの2行を読むだけで分かる。

only: [ :new, :create, ... ] と書かない理由：only 方式（許可リストの逆＝保護対象を列挙）だと、将来このコントローラにアクションを足したときリストへの追記を忘れた瞬間、そのアクションは誰でも実行できる。except 方式なら追記を忘れても新アクションは管理者限定に倒れる。ミスしたときに安全側に落ちる設計は、既存の認証 concern の「既定で全部ロック、公開ページだけ opt-out」と同じ思想。

ゲートの実行順序：before_action は登録順に走る。require_authentication は ApplicationController（の include）で登録済みなので、BooksController で足した require_admin より必ず先に実行される。だから require_admin の中では「ログイン済み」を前提にでき、未ログインユーザーの挙動（ログイン画面へ誘導）は従来と1ミリも変わらない。

情報源
  • Action Controller の概要 — フィルタ — before_action の only / except オプションと実行順序（親クラスのフィルタが先）の原典。
  • 検索キーワード: Rails before_action except only difference / Rails filter chain execution order inheritance / secure by default fail closed
03

ビューの出し分け — UXであってセキュリティではない

layouts/application・books/index・books/show の3ファイル。押せないボタンを見せない

ヘッダーの「本を登録」、一覧と詳細の「編集」「削除」を admin?（helper_method 経由）で出し分けた。これは「押しても拒否されるボタンを見せない」という UX 施策であって、防御の実体は 02 のコントローラガード。ここを混同しないのがこの機能の説明の核になる。

app/views/books/show.html.erb（操作ボタン部分）erb
<div class="card-actions justify-between mt-6">
  <%= link_to "一覧へ戻る", books_path, class: "btn btn-ghost" %>
  <% if admin? %>
    <div class="flex gap-2">
      <%= link_to "編集", edit_book_path(@book), class: "btn btn-outline" %>
      <%= button_to "削除", book_path(@book), method: :delete,
            class: "btn btn-error btn-outline",
            form: { data: { turbo_confirm: "「#{@book.title}」を削除します。よろしいですか？" } } %>
    </div>
  <% end %>
</div>
app/views/layouts/application.html.erb（ヘッダー）erb
<%= link_to "ログアウト", session_path, data: { turbo_method: :delete }, class: "btn btn-ghost btn-sm" %>
<% if admin? %>
  <%= link_to "本を登録", new_book_path, class: "btn btn-primary btn-sm" %>
<% end %>
なぜこの設計か（そうしないとどうなるか）

ビューの出し分けだけでは守れない：HTML にボタンが無くても、HTTP リクエストは curl や開発者ツールで誰でも作れる。実際に一般ユーザーのセッションで POST /books と DELETE /books/:id を直接叩く検証を行い、コントローラ側で拒否されることを確認済み（→ 04）。逆にコントローラガードだけでビューを出し分けないと、一般ユーザーに「押すと必ず拒否されるボタン」が見え続け、システムへの信頼を削る。両方やって初めて完成で、役割は「ビュー = UX、コントローラ = セキュリティ」と明確に分かれる。

判定源をコントローラの admin? に一本化：ビュー側に Current.user&.admin? を直書きする書き方もあるが、helper_method で公開した同じメソッドを使えば、判定ロジックを変えるとき（例：ロール制へ移行）に触る場所が concern の1メソッドだけで済む。

TDD の記録 — Red の中身まで確認した
実装前にテストを流し、7件が「認可が存在しない」ことをそのまま示す理由で失敗することを目視確認してから実装に入った。例：登録画面アクセスは「Expected response to be a <3XX: redirect>, but was a <200: OK>」、削除は「Book.count didn't change by 0, but by -1」。「Red になった」だけでなく失敗理由が期待どおりかまで見るのは、実装が無くても通ってしまう偽陽性テストを排除するため（CLAUDE.md のルール）。

05

想定問答 — 強みと、正直な弱点

レビュー会で聞かれそうな質問と回答の準備。強みは胸を張って、未実装は正直に + 「やるならこうする」をセットで

まず押さえる3つの強み

  • クローズ既定の認可except 方式なので、新アクションを足し忘れても管理者限定側に倒れる。認証 concern と同じ「安全側に落ちる」思想（→ 02）
  • 認証と認可の分離Authentication / Authorization を別 concern に。「誰なのか」と「何をしてよいか」を独立に変更・テストできる（→ 01）
  • 直叩きまで検証済みビューのボタン非表示だけでなく、curl による POST / DELETE 偽造がサーバー側で拒否されることを実機で確認（→ 04）

想定問答（クリックで回答を開く）

Q1ボタンを隠しているだけで、URL を直接叩かれたら操作できてしまうのでは？

強みできません。防御の実体はビューではなく before_action :require_admin で、一般ユーザーのセッションを使って POST /books（フォーム偽造）と DELETE /books/:id を curl で直接叩く検証を行い、どちらも root へリダイレクトされてレコードが変化しないことを確認済みです。ビューの出し分けは「押しても拒否されるボタンを見せない」UX 施策で、セキュリティはコントローラ層が担う、という役割分担で説明できます。

Q2Pundit や CanCanCan を使わなかったのはなぜですか？

説明できる選択権限が「admin か否か」の boolean 1つで、Pundit が本領を発揮する「リソース×アクション毎のポリシー」を張るほどの複雑さがないためです。このアプリは認証も gem なし（Rails 8 標準構成）なので、認可だけ gem にすると流儀が2つ混在します。ただし限界も把握していて、「所有者は自分の投稿だけ編集できる」のようなレコード単位の認可が増えたら Pundit 導入を検討する、という判断基準まで話せるようにしています。

Q3要件は「登録・削除」なのに、編集まで管理者限定にした理由は？

説明できる選択要件本文の「一般ユーザーは貸出・返却のみ可能」と、要件定義に含まれる画面遷移図の「書籍編集/削除（管理者のみ）」という記述を根拠に、書き込み系5アクション（new / create / edit / update / destroy）をまとめて管理者限定と解釈しました。仮に編集を一般ユーザーに開けると「削除はできないがタイトルを全部書き換えることはできる」という穴になり、「のみ可能」の趣旨に反すると判断しました。

Q4管理者ユーザーはどうやって作るんですか？

未実装管理者を昇格・降格する管理画面はありません。現状はシード（db/seeds.rb の admin@example.com）か rails console での user.update!(admin: true) のみです。会員登録フォームから admin を渡されても、strong parameters（permit(:name, :email_address, :password, :password_confirmation)）の対象外なのでマスアサインメントでの自己昇格はできません。やるなら：管理者専用の Admin::UsersController を作り、同じ require_admin ゲートを掛けた上でトグル操作を提供します。「最後の管理者を降格させない」ガードもセットで必要です。

Q5貸出履歴が残っている本を管理者が削除するとどうなりますか？

既存バグを発見削除されないのに「successfully destroyed」と表示されます。Book は has_many :rentals, dependent: :restrict_with_error なので履歴のある本の destroy は false を返しますが、BooksController#destroy が戻り値を確認せず常に成功メッセージを出すためです（本 PR の画面検証中に発見。認可とは独立した既存の問題なのでスコープを分け、PR 本文に記録しました）。やるなら：if @book.destroy で分岐し、失敗時は @book.errors.full_messages を alert で返します。「destroy は例外を投げず false を返す」ことを知っているか、が論点です。

Q6権限がないとき、403 Forbidden ではなくリダイレクトなのはなぜ？

説明できる選択このアプリの利用者はブラウザの人間だけなので、403 のエラー画面を見せて行き止まりにするより、トップへ戻して alert で理由を伝えるほうが次の行動につながると判断しました。API を提供するなら話が変わり、JSON リクエストには head :forbidden を返す format 分岐を require_admin に足すことになります。「誰がこのレスポンスを受け取るか」でステータスコードの正解が変わる、という整理です。

Q7「司書」のような中間ロールが増えたらどうしますか？

boolean の限界admin boolean は2値なので3ロール以上は表現できません。やるなら：role カラム（integer）＋ enum :role, { member: 0, librarian: 1, admin: 2 } へマイグレーションします。enum が admin? / librarian? を自動生成するので、concern の admin? 呼び出し面はそのまま生き、ビューとコントローラの変更は最小で済みます。判定源を admin? 1メソッドに一本化してあるのは、この移行を見越した作りです。

Q8require_admin の時点でログイン済みという保証はどこにありますか？

強みbefore_action の実行順序です。require_authentication は ApplicationController が include する Authentication concern で登録され、BooksController で足した require_admin より必ず先に走ります。そして認証をスキップする allow_unauthenticated_access only: [ :index, :show ] の対象は、require_admin の except: [ :index, :show ] と同じリスト。つまり「認証なしで require_admin に到達する経路」は構造的に存在しません。さらに admin? 自体も authenticated? && で始まるので、仮に前提が崩れても nil エラーではなく「権限なし」に安全に倒れます。
