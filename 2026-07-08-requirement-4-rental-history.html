概要

発展要件4（貸出履歴の閲覧）

スキーマ変更はなし。返却機能（必須要件4）を実装した時点で returned_at を更新するだけの状態遷移にしていたため、貸出履歴はテーブルに全件残っていた。今回はそれを読む経路（コントローラ・スコープ・ビュー）を追加しただけで実現している。

誰が見えるもの実現方法
一般ユーザー自分の貸出履歴のみCurrent.user.rentals（関連経由の絞り込み＝認可）
管理者全ユーザーの履歴＋借主名Rental.all ＋ ビュー側で admin? 時のみ「借主」列を表示

絞り込み（貸出中／返却済み／すべて）とソート（貸出日・返却日）は、UI/UX改善（発展要件5）で書籍一覧に導入した kaminari + sort_link + preserved_list_params のパターンをそのまま Rental にも展開したもの。ルーティングは books 配下にネストせず、既存のトップレベル resources :rentals（返却用の update が既にある）に index を足しただけ——貸出履歴は「本の下にぶら下がる情報」ではなく「ユーザー単位の一覧」だと考えたため。

01

認可とスコープ切り替え

「一般ユーザーは自分のみ／管理者は全員」の作り分け

新しい認可の仕組みは何も足していない。既存の admin?（app/controllers/concerns/authorization.rb、書籍のCRUD制限で既に使っていたもの）で、対象スコープをどちらの ActiveRecord::Relation にするか出し分けているだけ。

app/controllers/rentals_controller.rb（index・抜粋）ruby
def index
  @rentals = rental_scope.includes(:user, :book)
                         .with_status(params[:status])
                         .sorted(params[:sort], params[:direction])
                         .page(params[:page])
end

private

def rental_scope
  admin? ? Rental.all : Current.user.rentals
end
なぜこの設計か（そうしないとどうなるか）

Pundit などの認可 gem を導入しなかった。このアプリの認可要件は「管理者かどうかの二択」だけで、複雑なポリシーオブジェクトが要るほどの分岐はない。既存の require_admin/admin? と地続きの1行で足りるところに gem を足すと、判断基準が「この画面は concern、あの画面は Pundit」と割れてしまい、かえって読みづらくなる。

認可は if 文ではなく「関連から辿れるかどうか」で表現する。rental_scope が返すのが Current.user.rentals という関連である限り、他ユーザーの履歴はこの先の with_status／sorted／page にどう合成しても検索範囲そのものに入らない。返却機能（set_own_active_rental）で使ったのと同じイディオムをそのまま一覧にも展開した形。

02

絞り込み・並び替え

ステータス（貸出中／返却済み／すべて）フィルタと貸出日・返却日ソート

Rental モデルに with_status と sorted の2スコープを追加。どちらも書籍一覧（Book.sorted）で既に確立していた「許可リスト方式」をそのまま踏襲している。

app/models/rental.rb（ステータス絞り込み）ruby
STATUS_LABELS = { "active" => "貸出中", "returned" => "返却済み" }.freeze

scope :with_status, ->(status) {
  case status.to_s
  when "active" then active
  when "returned" then where.not(returned_at: nil)
  else all
  end
}
app/models/rental.rb（並び替え）ruby
SORTABLE_COLUMNS = %w[created_at returned_at].freeze

scope :sorted, ->(column, direction) {
  if SORTABLE_COLUMNS.include?(column.to_s)
    dir = %w[asc desc].include?(direction.to_s) ? direction.to_sym : :asc
    order(column.to_s => dir, id: :desc)
  else
    order(created_at: :desc, id: :desc)
  end
}
なぜこの設計か（そうしないとどうなるか）

params[:sort] を order に直接渡さない。もし order(params[:sort] => params[:direction]) と書いてしまうと、URL の sort パラメータをそのままカラム名として使うことになり、任意のカラムを指定されるSQLインジェクションの経路になる。SORTABLE_COLUMNS という許可リストに含まれる値だけを通し、それ以外は黙って安全なデフォルト（貸出日の新しい順）に落とすことで防いでいる。

id: :desc を第2キーに必ず添える。created_at だけでソートすると、同時刻に複数件あった場合の順序がDBの内部実装依存になり、ページネーションで1ページ目と2ページ目の境界にある行が重複したり消えたりし得る。一意な id を第2キーにすることで、同値があっても常に全順序が確定する。

ステータスと違い、ソートは case/when ではなく許可リスト＋orderにしている。ステータスは「貸出中／返却済み」という意味のある2値なので分岐で書いた方が読みやすいが、ソート対象カラムは今後増える可能性がある「データの並べ替えキー」なので、許可リストに1語足すだけで拡張できる形にしてある。

テストの見どころ
test/models/rental_test.rb では Rental.sorted("user_id", "asc") のように許可リスト外のカラムをわざと渡し、デフォルトの並び順にフォールバックすることを確認している。「防御コードが実際に効いている」ことをテストで裏付けている。

03

ビューとUI

レスポンシブ対応・daisyUI タブ・テスト容易性のための dom_id

書籍一覧（発展要件5）と同じ「スマホはカード、md以上はテーブル」の二重レイアウトを踏襲。加えて、各行に dom_id でID を振ることで、「どのユーザーの、どの貸出行が見えているか」をテストから厳密に検証できるようにした。

app/helpers/rentals_helper.rb（全文）ruby
module RentalsHelper
  def preserved_rental_list_params(overrides = {})
    {
      status: params[:status].presence,
      sort: params[:sort].presence,
      direction: params[:direction].presence
    }.merge(overrides).compact
  end

  def rental_sort_link(column, label)
    active = params[:sort] == column
    current_direction = params[:direction] == "desc" ? "desc" : "asc"
    next_direction = active && current_direction == "asc" ? "desc" : "asc"
    arrow = active ? (current_direction == "asc" ? " ▲" : " ▼") : ""

    link_to "#{label}#{arrow}",
            rentals_path(preserved_rental_list_params(sort: column, direction: next_direction)),
            class: "link link-hover"
  end
end
app/views/rentals/index.html.erb（タブと行ID・抜粋）ruby
<div class="tabs tabs-box w-fit mb-6">
  <%= link_to "すべて", rentals_path(preserved_rental_list_params(status: nil)),
        class: "tab #{'tab-active' if params[:status].blank?}" %>
  <% Rental::STATUS_LABELS.each do |value, label| %>
    <%= link_to label, rentals_path(preserved_rental_list_params(status: value)),
          class: "tab #{'tab-active' if params[:status] == value}" %>
  <% end %>
</div>

<tr id="<%= dom_id(rental) %>" class="hover">
  ...
</tr>
なぜこの設計か（そうしないとどうなるか）

dom_id(rental) で各行に rental_42 のようなIDを振ったのは見た目のためではなくテストのため。このアプリは Rails 標準の assigns（コントローラのインスタンス変数をテストから覗く仕組み）を使っておらず、既存のテストも response.body への assert_match で検証するスタイル。今回は同じ本を複数ユーザーが時期違いで借りている fixture（rentals(:one) と rentals(:two) は同じ本）があり、本のタイトルでは「誰の履歴か」を区別できない。dom_id でレコード単位のIDを振ることで、「このレンタルIDが body に含まれるか」というレコード粒度の厳密な検証ができるようになった。

daisyUI のクラス名は tabs-boxed ではなく tabs-box。daisyUI 4系の tabs-boxed は5系で tabs-box にリネームされており、実装前に app/assets/tailwind/daisyui.mjs を直接 grep してこのプロジェクトが daisyUI 5.5.23 であることを確認してから書いた。もし旧名のまま書いていたら、CSSはエラーにならず静かに無効なクラス名として無視されるため、レビュー時までタブの見た目だけが崩れたままになっていたはずの箇所。

想定問答 — 強みと、正直な弱点

レビュー会で聞かれそうな質問と回答の準備

まず押さえる3つの強み

  • 認可がコントローラ1行admin? ? Rental.all : Current.user.rentals だけで閲覧範囲が決まる。既存の認可パターンとの一貫性（→ 01）
  • スキーマ変更ゼロ返却を状態遷移として設計していたおかげで、履歴閲覧はモデルのスコープとビューの追加だけで実現できた（→ 00）
  • 同一データでも区別できるテスト同じ本を借りた別ユーザーの履歴を dom_id ベースで区別して検証。fixture の制約に負けない検証手段を選んだ（→ 03）

想定問答（クリックで回答を開く）

Q1返却期限や延滞は表示されますか？

未実装rentals テーブルに返却期限（due date）のカラムがなく、貸出日と返却日（returned_at）しか持っていないため、延滞判定はできません。要件4は「閲覧できる」ことのみを求めており延滞管理までは範囲に含めていないと判断し、今回は対象外にしました。やるなら：due_at カラムを追加し、Rental に scope :overdue, -> { active.where(due_at: ...Time.current) } のようなスコープを足せば、ステータスフィルタに「延滞中」を1つ増やすだけの拡張で済む設計になっています。

Q2管理者は特定の1ユーザーだけの履歴に絞り込めますか？

未実装（スコープ外と判断）実装前に「ステータスフィルタのみ」「ステータス＋ユーザー絞り込み」の2案で確認を取り、後者は要件4の範囲を超えるスコープ拡大だと判断してあえて見送りました。やるなら：rental_scope に admin? && params[:user_id].present? ? Rental.where(user_id: params[:user_id]) : ... を1段足し、管理者向けのユーザー選択セレクトをビューに追加するだけで実装できます。認可の1行にもう1条件足す形なので、既存の設計を壊さずに拡張できます。

Q3履歴が数千件に増えても一覧は重くなりませんか？

対策済みincludes(:user, :book) でN+1を防ぎ、kaminari の .page で1ページに出す件数を固定しているので、行数がどれだけ増えてもクエリ数・1リクエストで返す行数は増えません。書籍一覧（発展要件5）で確立したページネーション基盤をそのまま流用できた形です。

Q4なぜ books や users の配下にネストしなかったのですか？

説明できる選択貸出履歴は「特定の本に対する操作履歴」ではなく「ユーザーが今まで借りたものの一覧」なので、本単位のリソース（books/:id/rentals）にはなじみません。users 配下も検討しましたが、このアプリには users#show/index の前例がなく、ネストする必然性もなかったため、返却用に既にあったトップレベルの resources :rentals に index を足す形に決めました。

Q5一般ユーザー向けの一覧にも「借主」列を出さないのはなぜですか？

説明できる選択一般ユーザーの一覧は常に自分自身の履歴だけなので、借主名を出しても情報量がゼロです。admin? で列そのものを出し分け、一般ユーザーの画面をその分シンプルに保っています。
