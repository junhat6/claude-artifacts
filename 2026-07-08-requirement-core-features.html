01

ユーザー認証

必須要件 1（会員登録・ログイン・ログアウト、貸出返却はログイン済みのみ）＋発展（パスワードリセット）

Rails 8 の標準認証ジェネレータ（bin/rails generate authentication）と同じ構成で実装。Devise などの認証 gem を使わず、フレームワーク標準の部品だけで組んでいるのがポイント。最初は自前の session[:user_id] 方式で作り、セッション固定攻撃対策を挟んで、Rails 8 標準構成に移行するという3段階を踏んだ（経緯は docs/authentication.md に記録）。

中核：Authentication concern（クローズ既定）

app/controllers/concerns/authentication.rbruby
module Authentication
  extend ActiveSupport::Concern

  included do
    before_action :require_authentication   # 全アクションを既定で保護
    helper_method :authenticated?
  end

  class_methods do
    def allow_unauthenticated_access(**options)
      skip_before_action :require_authentication, **options
    end
  end

  private
    def require_authentication
      resume_session || request_authentication
    end

    def resume_session
      Current.session ||= find_session_by_cookie
    end

    def find_session_by_cookie
      Session.find_by(id: cookies.signed[:session_id]) if cookies.signed[:session_id]
    end

    def start_new_session_for(user)
      user.sessions.create!(user_agent: request.user_agent, ip_address: request.remote_ip).tap do |session|
        Current.session = session
        cookies.signed.permanent[:session_id] = { value: session.id, httponly: true, same_site: :lax }
      end
    end
end
なぜこの設計か

「既定で全部ロック、公開ページだけ opt-out」という向きが肝。ApplicationController が include するので、新しいコントローラを作った瞬間から認証必須になる。逆向き（既定オープンで保護したいページに before_action を足す）だと、足し忘れたページが認証なしで公開される事故が構造的に起こる。忘れたときに安全側に倒れる設計。

ログイン状態は cookie に user_id を直接入れるのではなく、sessions テーブルの行 ID を署名付き cookie に入れる。セッションが DB の行なので「パスワード変更時に全端末をログアウトさせる」（sessions.destroy_all）が1行で書ける。cookie は httponly（JS から読めない = XSS 対策）+ same_site: :lax（CSRF 緩和）+ 署名付き（改ざん検知）。

活用した Rails の機能

  • has_secure_password
    bcrypt でハッシュ化して password_digest に保存。authenticate / 確認入力の検証を自動提供。
  • normalizes7.1+
    保存時も検索時もメールを strip + downcase に正規化。「大文字で登録した人がログインできない」を防ぐ。
  • User.authenticate_by7.1+
    ユーザーが存在しない場合もダミーのハッシュ計算を行い応答時間を揃える（タイミング攻撃によるメールアドレス列挙の対策）。
  • rate_limit8.0+
    ログイン試行を 3分10回に制限（ブルートフォース対策）。gem なしのコントローラ組み込み機能。
  • ActiveSupport::CurrentAttributes
    Current.user でリクエスト中どこからでも現在ユーザーを参照。リクエスト毎に自動リセット。
  • パスワードリセットトークン8.1
    has_secure_password の既定 reset_token: true により password_reset_token（15分期限）が自動生成される。モデルに1行も書いてないのに動く理由として説明できるようにしておく。

パスワードリセットのメールは本番では Resend の HTTP API 経由（Fly.io/Render は SMTP がブロックされがちなため）。リセット成功時は @user.sessions.destroy_all で全端末ログアウト、また「そのメールアドレスが存在するか」を応答で漏らさないよう、存在有無にかかわらず同じ完了メッセージを返す。

02

書籍CRUD

必須要件 2（書籍の登録・閲覧・更新・削除。属性：タイトル・ISBN・出版年・出版社・著者）

books を resources による RESTful リソースとして実装。閲覧系（index / show）はゲスト可、書き込み系はログイン必須という切り分けを allow_unauthenticated_access の1行で表現している。

app/controllers/books_controller.rb（抜粋）ruby
class BooksController < ApplicationController
  allow_unauthenticated_access only: [ :index, :show ]  # 閲覧はゲスト可、他はログイン必須
  before_action :set_book, only: [ :show, :edit, :update, :destroy ]

  def index
    @books = Book.search(params[:q]).includes(:authors).order(created_at: :desc)
  end

  def create
    @book = Book.new(book_params)
    if @book.save
      redirect_to @book, notice: "Book was successfully created."
    else
      render :new, status: :unprocessable_entity   # Turbo は 422 を要求する
    end
  end

  def destroy
    @book.destroy
    redirect_to books_path, notice: "Book was successfully destroyed.", status: :see_other
  end

  private

  def book_params
    params.require(:book).permit(:title, :isbn, :published_year, :publisher,
                                 :new_author_name, author_ids: [])
  end
end
なぜこの設計か

RESTful に7アクションへ揃えると、URL・ヘルパ名・ビューの置き場所・テストの構成が全部 Rails の規約で決まり、読む人が迷わない（「設定より規約」）。独自アクションを生やすほど、この予測可能性が失われていく。

strong parameters は「許可した属性しかマスアサインメントさせない」ホワイトリスト。author_ids: [] のように配列は明示が必要で、書き忘れると Rails はエラーにせず黙って捨てるため「著者だけ保存されない」という気づきにくいバグになる。

保存失敗時の status: :unprocessable_entity（422）と削除後の status: :see_other（303）は Turbo Drive の要求仕様。Turbo はフォーム送信への 200 応答を正しく処理できないので、Rails 7 以降は「失敗 = 422 で再描画、成功 = リダイレクト」が規約になっている。

活用した Rails の機能

  • resources :books
    7アクションの RESTful ルーティング。root "books#index" で一覧をトップに。
  • form_with model:
    new / edit で同じ _form パーシャルを共有。モデルの状態から POST / PATCH を自動判別。
  • includes(:authors)
    一覧で各書籍の著者を表示するため eager load して N+1 を回避（bullet gem で検知体制も）。
  • turbo_confirm
    削除ボタンに data: { turbo_confirm: ... } で確認ダイアログ。JS を書かずに済む。

多対多リレーション（書籍 ↔ 著者）

必須要件 3（多対多）。設計判断は docs/design-decisions-authors.md に ADR として文書化

中間モデル BookAuthor を明示的に作る has_many :through 方式。中間モデルを持たない has_and_belongs_to_many ではなくこちらを選んだ理由まで含めて ADR に記録してある（レビューで最も説明しやすい部分）。

app/models/book.rb ・ author.rb ・ book_author.rb（関連部分）ruby
class Book < ApplicationRecord
  has_many :book_authors, dependent: :destroy            # 本を消せば紐付けも道連れ
  has_many :authors, through: :book_authors
end

class Author < ApplicationRecord
  has_many :book_authors, dependent: :restrict_with_error # 本が紐付く著者は消せない
  has_many :books, through: :book_authors
  validates :name, presence: true, uniqueness: true
end

class BookAuthor < ApplicationRecord
  belongs_to :book
  belongs_to :author
  validates :author_id, uniqueness: { scope: :book_id }   # アプリ層の重複防止
  # + DB 層にも (book_id, author_id) の複合ユニーク index（二層防御）
end

フォームからの著者指定：既存選択 + 新規追加の両対応

フォームには「既存著者の複数選択（author_ids: []）」と「新しい著者名のテキスト入力」を併設。新規著者名は DB カラムではない仮想属性として受ける。

app/models/book.rb（仮想属性と取り込み）ruby
attr_reader :new_author_name
before_save :attach_new_author

def new_author_name=(value)
  @new_author_name = value.to_s.strip.presence  # 空白だけの入力は nil に潰す
end

def attach_new_author
  return if new_author_name.blank?

  author = Author.find_or_create_by!(name: new_author_name)
  authors << author unless authors.exists?(author.id)
end
なぜこの設計か

habtm ではなく has_many :through：中間テーブルがモデルとして存在するので、将来「著 / 訳 / 監修」のような役割カラムを足したくなってもテーブルを作り直さずに済む。habtm は中間に属性を持てないため、必要になった時点で移行作業が発生する。

ユニーク制約の二層防御：アプリ層の uniqueness は分かりやすいエラーメッセージ用、DB 層の複合ユニーク index は並行リクエストでも重複を通さない最終防衛線。アプリ層だけだと同時送信で重複が入り得る（04 の貸出と同じ理屈）。

dependent の非対称：Book 側は :destroy（本を消したら紐付けは無意味なので道連れ）、Author 側は :restrict_with_error（著者削除画面が無いので、誤削除からデータを守る保守的な選択）。「同じ関連でも向きによって削除ポリシーは変わる」の実例。

find_or_create_by!：同名著者の重複作成を防ぐ（authors.name にもユニーク index）。同姓同名の別人はこの課題ではスコープ外、と割り切りも ADR に明記。

04

貸出と三層防御

必須要件 4（貸出。貸出中の書籍は他ユーザーが借りられない）— この課題の技術的ハイライト

「貸出中の本は借りられない」をモデル・DB・コントローラの三層で守る。単なるバリデーション1個ではなく、同時リクエスト（レースコンディション）まで考慮した構成。

第1層モデルのバリデーション役割：UX（丁寧なエラー文）

validate :book_must_be_available, on: :create — 通常ケースの二重貸出を「貸出中のため借りられません」という分かりやすいメッセージで弾く。

同時に2リクエストが来ると両方「貸出中なし」と判定してすり抜ける（check-then-act の隙間）
第2層DB の部分ユニーク index役割：整合性の最終防衛線

returned_at IS NULL の行に限って book_id を一意にする。片方の INSERT は物理的に失敗し、DB が二重貸出を構造的に禁止する。

制約違反は ActiveRecord::RecordNotUnique 例外として上がってくる
第3層コントローラの rescue_from役割：例外を UX に変換

500 エラー画面ではなく「ちょうど貸出中になったため借りられませんでした。」というフラッシュに変換して書籍詳細へ戻す。

db/migrate/20260702151447_create_rentals.rb（第2層）ruby
# returned_at が NULL（貸出中）の行に限り book_id を一意にする
add_index :rentals, :book_id, unique: true,
  where: "returned_at IS NULL",
  name: "index_rentals_on_book_id_when_active"
app/models/rental.rb（第1層・全文）ruby
class Rental < ApplicationRecord
  belongs_to :user
  belongs_to :book

  scope :active, -> { where(returned_at: nil) }

  validate :book_must_be_available, on: :create

  private

  def book_must_be_available
    return if book.blank?
    return unless Rental.active.exists?(book_id: book_id)

    errors.add(:book, "は貸出中のため借りられません")
  end
end
app/controllers/rentals_controller.rb（第3層・抜粋）ruby
class RentalsController < ApplicationController
  rescue_from ActiveRecord::RecordNotUnique, with: :book_already_rented

  def create
    @rental = @book.rentals.new(user: Current.user)
    if @rental.save
      redirect_to @book, notice: "「#{@book.title}」を借りました。"
    else
      redirect_to @book, alert: @rental.errors.full_messages.to_sentence
    end
  end

  private

  def book_already_rented
    redirect_to @book, alert: "ちょうど貸出中になったため借りられませんでした。"
  end
end
なぜ三層必要か（そうしないとどうなるか）

バリデーションの実体は「SELECT で存在確認 → OK なら INSERT」という2ステップで、この間には必ず時間の隙間がある。2人が同時に「借りる」を押すと、両方の SELECT が「貸出中なし」を返し、両方 INSERT されてしまう。Rails ガイド自身が「uniqueness 検証は競合状態を防げないので DB 制約を併用せよ」と明記している問題。

逆に DB 制約だけにすると、通常ケースでもユーザーに見えるのは例外由来の無骨なエラーになる。各層の役割が違う（モデル=UX、DB=整合性、コントローラ=例外の翻訳）から三層とも要る、と説明できるのが強い。

「部分」index である点も重要：単なる book_id ユニークだと返却済み履歴が2行目から入らなくなる。WHERE returned_at IS NULL で「現役の貸出だけ」に制約を絞ることで、履歴の蓄積と排他制御を両立している。

テストの見どころ
test/models/rental_test.rb では save!(validate: false) でわざと第1層を迂回し、assert_raises(ActiveRecord::RecordNotUnique) で第2層が単体で機能することを検証している。「バリデーションを通らない経路でも DB が守る」ことをテストで証明済み、と言える。

05

返却

必須要件 4（返却ができる）

返却は「レコードの削除」ではなく returned_at に時刻を入れる状態遷移として実装。貸出の記録がそのまま履歴として残る。

app/controllers/rentals_controller.rb（返却部分）ruby
before_action :set_own_active_rental, only: :update

def update
  @rental.update!(returned_at: Time.current)
  redirect_to @rental.book, notice: "「#{@rental.book.title}」を返却しました。"
end

private

def set_own_active_rental
  @rental = Current.user.rentals.active.find(params[:id])  # 本人の貸出中レコードだけ対象
end
なぜこの設計か

Current.user.rentals.active.find(...) が認可そのものになっている。「全体から find して所有者かどうか if 文で比較」する書き方だと、比較を書き忘れた瞬間に他人の貸出を返却できてしまう。ログインユーザーの関連からたどれば、他人のレコードはそもそも検索範囲に入らず RecordNotFound（= 404）になる。存在の有無すら漏らさない、Rails の定番認可イディオム。

削除ではなく更新にしたことで、発展要件の「貸出履歴の閲覧」にテーブル設計変更なしでそのまま進める。DELETE だと履歴要件が出た時点で設計やり直しになる。

update!（bang 付き）なのは、set_own_active_rental を通過した時点で失敗する正当な理由がないから。ここで失敗するならバグなので、握り潰さず例外にして気づけるようにしている。

書籍検索

必須要件 5（一覧表示・タイトル / 著者名での検索）

検索ロジックはコントローラではなく モデルの scope に置き、1つの LIKE クエリでタイトルと著者名を横断検索する。短いコードに検索実装の定番の罠への対策が3つ詰まっている。

app/models/book.rb（scope :search）ruby
scope :search, ->(query) {
  return all if query.blank?

  q = "%#{sanitize_sql_like(query)}%"
  left_joins(:authors)
    .where("books.title LIKE :q OR authors.name LIKE :q", q: q)
    .distinct
}
app/controllers/books_controller.rb（利用側）ruby
def index
  @books = Book.search(params[:q]).includes(:authors).order(created_at: :desc)
end
なぜこの書き方か（3つの罠と対策）

① sanitize_sql_like — プレースホルダ（:q）は SQL インジェクションを防ぐが、% や _ は「値として安全」なままワイルドカードとして解釈される。ユーザーが % を検索したら文字通りの % にマッチしてほしいので、別途エスケープが要る。

② left_joins — INNER JOIN（joins）だと著者が紐付いていない本が結果から消え、タイトル検索でもヒットしなくなる。LEFT OUTER JOIN なら著者ゼロの本も検索対象に残る。

③ distinct — 共著（1冊に著者2人）を JOIN すると行が2行に複製され、一覧に同じ本が2回出る。distinct で畳む。テストにも「共著でも重複しない」ケースがある。

スコープをモデルに置くのは、Book.search(q).includes(...) のように他のクエリと合成できるため。コントローラに where を直書きすると再利用も単体テストもできなくなる。

Fly.io + SQLite（Rails 8 の「gem レス」構成）

本番は Fly.io（東京リージョン）で、DB は本番も SQLite。永続ボリューム /data に DB ファイルを置き、primary / cache / queue / cable の4つの SQLite ファイルに分けている。キャッシュは Solid Cache、ジョブは Solid Queue、Action Cable は Solid Cable が担い、Redis も PostgreSQL も不要という Rails 8 が推す最小構成。

  • HTTP は Thruster 経由（bin/thrust bin/rails server）、アイドル時はマシン自動停止でコスト最小化、ヘルスチェックは GET /up

想定問答（クリックで回答を開く）

Q1同時に2人が「借りる」ボタンを押したらどうなりますか？

強み片方だけが借りられます。バリデーションは両者をすり抜け得ますが（SELECT と INSERT の間に隙間があるため）、returned_at IS NULL の行だけ book_id を一意にする部分ユニーク index が後着の INSERT を弾き、RecordNotUnique を rescue_from で捕まえて「ちょうど貸出中になったため借りられませんでした」と案内します。save!(validate: false) でバリデーションを迂回して DB 層単体を検証するテストも書いてあります。

Q2管理者だけが書籍を登録・削除できるようになっていますか？

未実装正直に言うと、users.admin カラム（boolean）は DB に用意しましたが、認可ロジックは未実装です。現状はログインさえすれば誰でも書籍を CRUD でき、編集・削除ボタンも全ユーザーに表示されます。docs/crud-matrix-and-screens.md の CRUD マトリクスには「書籍管理=管理者」と書いており、ドキュメントと実装が乖離している状態です。

やるなら：Authentication concern と同じパターンで require_admin を作り、BooksController の書き込み系アクションに before_action で掛け、ビューのボタンも Current.user&.admin? で出し分けます。認証（誰であるか）と認可（何ができるか)を別レイヤーにする、という整理まで話せると良いです。

Q3書籍が150冊あると一覧が全件表示されますが、重くないですか？

未実装ページネーションは未実装です（発展要件 5）。includes(:authors) で N+1 は防いでいるのでクエリ数は一定ですが、行数はデータ量に比例して増えます。やるなら pagy か kaminari を導入し、Book.search(params[:q]).includes(:authors).page(params[:page]) のようにスコープの合成に1メソッド足すだけで済む設計にはなっています。

Q4貸出履歴はどこで見られますか？

未実装履歴閲覧画面（発展要件 4）は未実装です。ただし返却を「削除」ではなく returned_at の更新にしたため、データとしての履歴は既に全件残っています。rentals#index を追加して Current.user.rentals.includes(:book).order(created_at: :desc) を出すだけで実装でき、テーブル設計の変更は不要です。設計が先回りできている点はアピールポイントにできます。

Q5books テーブルに NOT NULL 制約がないのはなぜですか？

改善余地books の title などの必須チェックはモデルの presence バリデーションのみで、DB の NOT NULL 制約を付けていません。update_column や SQL 直叩きなどバリデーションを通らない経路では NULL が入り得ます。rentals では「アプリ層 + DB 層の二層防御」を実践したので、同じ考え方を books にも適用して NOT NULL を後付けするのが次の改善です（users / authors / 中間テーブルには NOT NULL があります）。

Q6同じ本を複数冊登録できますか？（在庫管理）

未実装発展要件 2 は未対応で、現状「1 ISBN = 1冊」です（isbn にユニーク index）。やるなら「書誌（books）」と「現物（book_items）」を分ける設計にし、貸出の外部キーを book_id から book_item_id に付け替えます。部分ユニーク index の対象も book_item_id に変わるだけで、三層防御の考え方はそのまま流用できます。requirements/er_diagram.md に発展版の ER 図として下書きがあります。

Q7なぜ本番 DB が PostgreSQL ではなく SQLite なんですか？

説明できる選択Rails 8 が SQLite の本番運用を公式にサポートし（Solid Queue / Cache / Cable も SQLite で動く）、この規模のアプリでは運用部品が少ないほど堅牢だと判断しました。直接のきっかけは Render 無料枠の Postgres が30日で失効する制約で、Fly.io の永続ボリュームに SQLite を置く構成へ移行しました（経緯は docs/deployment-fly-io.md に記録）。要件の「RDBMS（PostgreSQL）」に対しては、SQLite も RDBMS であること・移行判断を文書化していることを添えて説明します。マルチマシンへスケールする場合は Postgres に戻す判断になる、という限界も把握しています。

Q8認証を gem（Devise など）にしなかったのはなぜですか？

強み学習目的の課題なので、Rails 8 標準の認証ジェネレータ構成を選びました。Devise はブラックボックスになりがちですが、標準構成なら has_secure_password → Session モデル → 署名 cookie という流れを全行説明できます。セッション固定攻撃対策・rate_limit・authenticate_by のタイミング攻撃耐性・パスワードリセットの15分トークンまで、何がどこで守られているか把握しています（→ 01）。
